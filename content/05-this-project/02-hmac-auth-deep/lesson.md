# HMAC-аутентификация глубже

На предыдущем уроке мы коснулись HMAC поверхностно: «Telegram подписывает данные, сервер проверяет». Этого хватает, чтобы запустить приложение, но не хватает, чтобы на собесе объяснить, **почему это безопасно**, **что атакующий может и не может**, и **где проходят границы метода**. В этом уроке — глубже.

> Главная мысль: HMAC решает строго одну задачу — **подтвердить, что данные не были изменены тем, у кого нет секретного ключа**. Всё, что вне этой задачи (шифрование, защита от повтора, защита от утечки секрета) — отдельные проблемы со своими решениями.

## Подпись ≠ шифрование

Это первое, что путают на собесах.

**Шифрование (encryption):**
- Превращает читаемые данные в нечитаемые.
- Чтобы прочитать — нужен ключ.
- Цель: **конфиденциальность**.

**Подпись (HMAC):**
- Данные остаются читаемыми.
- К ним прикладывается короткий код — подпись.
- Цель: **целостность** (не изменены) и **подлинность** (источник имел ключ).

Когда Telegram передаёт `initData` через `window.Telegram.WebApp`, ты можешь открыть DevTools и прочитать всё: `user_id`, `first_name`, `auth_date`. Никакого шифрования. Но если ты попробуешь поменять `user_id=123` на `user_id=456`, `hash` перестанет совпадать с тем, который пересчитает сервер.

> На собесе: «Можно ли через HMAC сделать конфиденциальную передачу?» — Нет. Для конфиденциальности нужен другой инструмент: HTTPS-канал (TLS), симметричное шифрование (AES), асимметричное (RSA). HMAC ортогонален.

## Что такое hash и почему его недостаточно

**Hash-функция** (например, SHA-256) — однонаправленная: легко посчитать `hash(data)`, невозможно из `hash(data)` восстановить `data`. Двух одинаковых hash для разных данных «практически не бывает» — это коллизия, для SHA-256 шанс ~ 1 на 2^128.

Кажется, можно было бы просто прикладывать `sha256(data)` и проверять. Но:

```
data = "user_id=123"
hash = sha256("user_id=123") = "abc..."
```

Атакующий тоже знает `data`, тоже умеет считать SHA-256. Подменяет:

```
fake_data = "user_id=999"
fake_hash = sha256("user_id=999") = "xyz..."
```

И посылает. Сервер видит: hash совпадает с тем, что он сам пересчитает. Принимает. Проблема: **обычный hash не требует секрета**, поэтому подделать его может кто угодно.

## HMAC: hash с секретом

**HMAC = Hash-based Message Authentication Code.** Это hash, в который «вмешан» секретный ключ:

```
hmac(key, data) = sha256( (key XOR opad) || sha256( (key XOR ipad) || data ) )
```

Не нужно помнить формулу — нужно понять идею: без знания `key` невозможно посчитать `hmac(key, data)`. У атакующего есть `data` и `hmac`, но нет `key` → подделать не может.

В нашем коде:
```ts
crypto.createHmac("sha256", secret).update(dataString).digest("hex");
```

`secret` — это **производный ключ** от токена бота. О деривации — ниже.

## Деривация ключа: почему не используют BOT_TOKEN напрямую

В Telegram-документации формула:

```
secret_key = HMAC_SHA256(key="WebAppData", data=BOT_TOKEN)
hash = HMAC_SHA256(key=secret_key, data=data_check_string)
```

То есть HMAC применяется **дважды**. Зачем такая сложность?

**Принцип:** один секрет — одна цель. BOT_TOKEN используется для разных вещей: вызов Bot API, авторизация webhook, валидация Mini App. Если бы во всех местах использовался один и тот же ключ, утечка из одного места компрометировала бы все.

Telegram производит **разные** ключи из одного `BOT_TOKEN` для разных контекстов:
- Для Mini App: `HMAC("WebAppData", BOT_TOKEN)`.
- Для Login Widget: другая константа.

Это паттерн **key derivation** (вывод ключа). Реальные библиотеки (HKDF, PBKDF2) делают это формализованно — Telegram сделал упрощённую версию.

> На собесе: «Зачем дважды HMAC?» — Чтобы из одного master-key получить контекстно-специфичные sub-keys. Это **separation of concerns на уровне криптографии**.

## Сборка data_check_string

Перед тем как считать HMAC, серверу нужно собрать **те же байты**, которые подписал Telegram. Малейшее расхождение (порядок параметров, лишний пробел) — hash не совпадёт.

Алгоритм:
1. Взять все параметры из initData, **исключая `hash`**.
2. Отсортировать по имени параметра (по алфавиту).
3. Склеить через `\n` в формате `key=value`.

```
auth_date=1700000000
query_id=AAH...
user={"id":123,"first_name":"Лиза"}
```

Почему сортировка важна: HMAC чувствителен к порядку байт. `"a=1\nb=2"` и `"b=2\na=1"` дают разные hash. Без договорённости о порядке сервер не смог бы воспроизвести подпись.

## Timing attack: почему `===` опасен

Серверный код мог бы выглядеть так:

```ts
if (computedHash === receivedHash) return ok();
```

Не делай так. JavaScript-сравнение строк работает посимвольно и **останавливается** на первом несовпадении. Это создаёт **timing side-channel**:

```
hash = "abc1234..."
attacker tries "a0000000..."  → mismatch at index 1 → 0.1 мс
attacker tries "ab000000..."  → mismatch at index 2 → 0.2 мс
attacker tries "abc00000..."  → mismatch at index 3 → 0.3 мс
```

Измеряя время ответа, атакующий перебирает hash **посимвольно**, а не целиком. Вместо `2^256` попыток ему нужно `16 × 64 = 1024` для hex-строки.

**Защита: `crypto.timingSafeEqual`** — тратит одинаковое время независимо от того, где первое расхождение. Внутри: XOR всех байт + проверка, что результат `0`. Идёт от начала до конца всегда.

```ts
const a = Buffer.from(computedHash, "hex");
const b = Buffer.from(receivedHash, "hex");
if (a.length !== b.length) return fail(); // длина — не секрет
if (!crypto.timingSafeEqual(a, b)) return fail();
```

> На собесе: «Что такое timing attack и как защититься?» — Side-channel атака, использующая разницу во времени выполнения. Защита: constant-time сравнение. Тот же паттерн используется при сравнении паролей, JWT, API-ключей.

## Replay attack: HMAC её не решает

Допустим, атакующий подсмотрел запрос с валидным `initData` (например, в публичном Wi-Fi без HTTPS, или через злую расширение). HMAC валиден — данные не изменены. Но это **не его** initData. Если он повторит запрос — сервер примет.

Это **replay attack** — повтор валидного запроса. HMAC сам по себе её не решает, потому что подпись остаётся валидной.

**Защита — `auth_date`:** Telegram включает в подписанные данные timestamp. Сервер проверяет: если `auth_date` старше N часов — отклоняет, даже при валидной подписи.

```ts
const MAX_AGE_HOURS = 24;
if (Date.now() / 1000 - authDate > MAX_AGE_HOURS * 3600) {
  return fail("initData expired");
}
```

Окно компромисса:
- Слишком короткое (5 минут): юзер открыл бота утром, вернулся вечером — нужна повторная авторизация.
- Слишком длинное (неделя): украденный initData валиден неделю.

24 часа — типичный компромисс для Mini App.

> Дополнительная защита для критичных операций (платёж, удаление): хранить на сервере **nonce** (одноразовый идентификатор запроса) и отклонять повторы. Для нашего приложения это overkill.

## Что хранится на сервере

Stateless подход означает: сервер не хранит сессии. Но кое-что у него всё-таки есть:

| Что | Где | Зачем |
|---|---|---|
| `BOT_TOKEN` | env-переменная | Источник секретного ключа |
| Запись пользователя в БД | таблица `users` | Прогресс, стрики, XP — нужно знать, кто это |
| Никаких токенов сессии | — | Каждый запрос приходит с initData, проверяется заново |

**Важно:** `BOT_TOKEN` **никогда не уезжает на клиент**. Если бы уехал — клиент мог бы сам подписать любые данные. Он живёт только в серверных env-переменных и в `process.env.BOT_TOKEN` на серверном коде.

> На Vercel есть разделение: env-переменные с префиксом `NEXT_PUBLIC_` экспортируются в браузер, остальные — только на сервер. `BOT_TOKEN` — без префикса, значит сервер-онли.

## Безопасность по слоям

HMAC решает **одну** задачу. Реальная безопасность приложения — комбинация нескольких слоёв:

| Угроза | Защита |
|---|---|
| Подмена данных юзером | HMAC-подпись (initData) |
| Перехват трафика | HTTPS (TLS на уровне канала) |
| Подбор подписи через timing | `timingSafeEqual` |
| Повтор валидного запроса | `auth_date` + max age |
| Утечка BOT_TOKEN | Хранение в env-переменных, не в Git |
| Утечка через ошибки логирования | Не логировать initData / BOT_TOKEN |
| Подделка клиентского состояния | Серверная валидация (`score > 100` — отклонить) |

Никакой отдельный слой не достаточен. На собесе любят спрашивать «а что если...» — отвечай через слои.

## Что важно вынести

- **HMAC ≠ шифрование.** Подпись подтверждает целостность и подлинность, не скрывает данные.
- **Hash без секрета** подделать тривиально, поэтому HMAC включает ключ.
- **Деривация ключа** (двойной HMAC у Telegram) — разделение контекстов: разные sub-keys для разных задач из одного master.
- **data_check_string** должна собираться **байт в байт** так же, как у источника подписи — сортировка параметров критична.
- **`timingSafeEqual`** защищает от timing attack — это constant-time сравнение, обязательное для криптографических проверок.
- **HMAC не защищает от replay.** Нужен `auth_date` + проверка max age. Для критичных операций — nonce.
- **BOT_TOKEN — серверный секрет.** Никогда не уезжает на клиент. Утечка = полная компрометация.
