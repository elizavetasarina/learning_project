# Lazy reset стриков и дневного XP — без cron

В этом проекте есть две метрики, которые «обнуляются» по календарю: **дневной XP** (сбрасывается в полночь) и **стрик** (растёт или ломается по дням). Классическое решение — **cron-задача** в 00:00. Я выбрала другой путь: **lazy reset** — работа делается только когда юзер реально пришёл. Этот урок — про сам паттерн, мой код в `lib/auth.ts`, и почему это и хорошее **собесовое решение**, и идеальный пример **lazy evaluation** в архитектуре.

> Главная мысль: если у системы есть «событие по календарю» (стрик, дневная цель, окончание подписки), не обязательно нужен cron. Часто достаточно проверять состояние **при следующем взаимодействии** — это дешевле, точнее и работает с любой таймзоной юзера.

## Наивный подход: cron

Очевидное решение:

```
00:00 UTC → cron-задача:
  UPDATE users SET daily_xp = 0, daily_xp_date = NOW();
  // плюс пересчёт стриков
```

**Что в нём плохо для нашей системы:**

1. **Работа делается для всех**, включая тех, кто не зайдёт сегодня. На 10 000 юзеров — 10 000 UPDATE для 100 активных.
2. **Один cron — одна таймзона**. У юзера в МСК «новый день» в 03:00 UTC, у юзера в Самоа — в 13:00 UTC. Один cron не угодит всем.
3. **Сбой cron = пропуск дня**. Vercel Free Tier допускает «приблизительное» время и может пропустить запуск. В нашей логике пропуск дня = неверные стрики у всех.
4. **Сложность инфраструктуры**. Cron — отдельный механизм деплоя, отдельные логи, отдельные секреты. Не для MVP.

## Lazy reset: ничего не делаем заранее

**Идея:** не сбрасывать счётчики в фоне. Сбрасывать их **в момент**, когда юзер сделал первый запрос за новый день — внутри `authenticateRequest`.

Алгоритм:
1. Юзер открыл Mini App / сделал любой API-запрос.
2. Аутентифицируем (HMAC валидация initData).
3. Сравниваем «текущий день в tz юзера» с «днём последней активности».
4. Если день сменился — обнуляем `dailyXp`, апдейтим стрик, обновляем `lastActivityAt`.

Никакого cron. Никаких background-задач. Только реакция на пользовательскую активность.

## Код в `lib/auth.ts`

```ts
if (Date.now() - lastActivity > ONE_HOUR) {
  // считаем «новый день» в таймзоне юзера, не UTC
  const today = startOfDayInTz(now, user.timezone);
  const lastDay = user.lastActivityAt
    ? startOfDayInTz(user.lastActivityAt, user.timezone)
    : null;

  // ─── стрик ───
  let newStreak = user.currentStreak;
  if (!lastDay) {
    newStreak = 1;  // первый визит
  } else {
    const diffDays = daysBetweenInTz(lastDay, today, user.timezone);
    if (diffDays === 0) {
      // тот же день — ничего не меняется
    } else if (diffDays === 1) {
      newStreak = user.currentStreak + 1;  // следующий день
    } else {
      newStreak = 1;  // пропуск 2+ — сброс
    }
  }

  // ─── dailyXp reset ───
  const dailyXpDay = user.dailyXpDate
    ? startOfDayInTz(user.dailyXpDate, user.timezone)
    : null;
  const isNewDay =
    !dailyXpDay || daysBetweenInTz(dailyXpDay, today, user.timezone) > 0;

  // fire-and-forget update в БД (не блокируем ответ)
  prisma.user.update({
    where: { id: user.id },
    data: {
      lastActivityAt: now,
      currentStreak: newStreak,
      longestStreak: Math.max(newStreak, user.longestStreak),
      ...(isNewDay && { dailyXp: 0, dailyXpDate: now }),
    },
  }).catch(() => {});

  // мутация локального объекта для актуального ответа
  user.currentStreak = newStreak;
  user.dailyXp = isNewDay ? 0 : user.dailyXp;
  user.lastActivityAt = now;
}
```

Несколько важных приёмов в этом блоке.

## Приём 1: throttle 1 час

```ts
if (Date.now() - lastActivity > ONE_HOUR) {
```

Без этого условия: при открытии главной клиент делает ~3 запроса параллельно (`/api/progress`, `/api/reviews/due`, etc.). Каждый бы пересчитывал стрики и писал в БД. 3 UPDATE'а вместо одного.

С throttle — **максимум 1 UPDATE в час** на юзера. Дополнительные запросы внутри часа возвращают уже посчитанное состояние из объекта `user`.

**Tradeoff:** стрик в UI может «отставать» до часа от факта. Если юзер заходит в 23:30 → стрик считается. В 00:01 (новый день в его tz) → не пересчитается, потому что прошло < часа. В 00:30 — пересчитается, стрик +1. Для UX незаметно.

## Приём 2: Conditional spread

```ts
data: {
  lastActivityAt: now,
  currentStreak: newStreak,
  longestStreak: Math.max(newStreak, user.longestStreak),
  ...(isNewDay && { dailyXp: 0, dailyXpDate: now }),
}
```

Идиома `...(condition && { fields })`:
- `condition === true` → `...({fields})` → ключи добавятся
- `condition === false` → `...false` → пропуск (spread от `false` в объекте — no-op)

Альтернатива:
```ts
const data = { lastActivityAt: now, currentStreak: newStreak, ... };
if (isNewDay) {
  data.dailyXp = 0;
  data.dailyXpDate = now;
}
```

Conditional spread — короче, иммутабельно (создаём один объект, не модифицируем по шагам). На собесе по TS это **идиома мидл+** уровня.

## Приём 3: Fire-and-forget

```ts
prisma.user.update({ ... }).catch(() => {});
```

**Нет `await`.** Запрос летит в БД, мы продолжаем без ожидания. `.catch(() => {})` — глотаем потенциальную ошибку, чтобы не было unhandled rejection.

**Почему так:**
- UPDATE — это «обновить статистику». Если он упадёт — не критично, обновится при следующем запросе (lazy паттерн самокомпенсирующийся).
- Ждать UPDATE 50-100мс на каждом запросе бессмысленно, если можно ответить юзеру сразу.

**Цена:** если БД упала в момент обновления, новое значение не сохранится. Юзер увидит «локальное» значение в ответе, но при следующем запросе оно вернётся к старому из БД. Терпимо.

## Приём 4: Мутация локального user перед return

```ts
user.currentStreak = newStreak;
user.dailyXp = isNewDay ? 0 : user.dailyXp;
user.lastActivityAt = now;
```

Почему: `prisma.user.update` — fire-and-forget, БД ещё не обновилась. Но мы хотим вернуть юзеру **актуальный** ответ, чтобы UI показал новый стрик/dailyXp сразу.

Решение — мутировать локальный объект `user` (который вернётся из `authenticateRequest`). Когда API-роут построит JSON-ответ — там будут уже новые значения.

Без этой мутации юзер бы увидел старые значения, и только после перезагрузки страницы — новые (когда БД-апдейт завершился, GET получил бы новое).

## Приём 5: TZ юзера через `Intl.DateTimeFormat`

Главный спор: что значит «новый день»? Для юзера в МСК — 00:00 по МСК. Для юзера в Самоа — 00:00 по их времени.

Решение — клиент шлёт IANA-таймзону в заголовке `X-Timezone`, сервер сохраняет в `users.timezone`, и вся логика дат идёт через **локальные сутки юзера**.

В `lib/datetime.ts`:
```ts
export function startOfDayInTz(date: Date, tz: string | null): Date {
  if (!tz) {
    // fallback на UTC
    const d = new Date(date);
    d.setUTCHours(0, 0, 0, 0);
    return d;
  }
  // получаем offset tz в указанной дате через Intl
  const offsetMin = getTzOffsetMinutes(date, tz);
  const shifted = new Date(date.getTime() + offsetMin * 60_000);
  shifted.setUTCHours(0, 0, 0, 0);
  return new Date(shifted.getTime() - offsetMin * 60_000);
}
```

Без библиотек — нативный `Intl.DateTimeFormat`. Это **important** для собеса: рекрутер любит спросить «как обрабатываете таймзоны без либ?» — есть готовый ответ.

## Tradeoffs

| Что | Cron | Lazy reset |
|---|---|---|
| Кому делается работа | Всем юзерам | Только активным |
| Инфраструктура | Отдельный механизм | Часть основного кода |
| Точность времени | Зависит от точности cron | Точно при первом запросе после смены дня |
| Таймзоны | Один cron — одна tz | Каждый юзер в своей tz |
| Сбой = пропуск | Да, целый день | Нет, следующий запрос всё пересчитает |
| Сложность кода | Простой UPDATE | Логика 4-х веток (тот же день, +1, +2..., null) |

**Минус lazy:** логика сложнее. У cron всё в одном месте, у lazy — внутри auth-флоу.

**Когда выбрать cron:**
- Юзеров много, активных мало, но нужно регулярно «трогать» неактивных (для аналитики, для рассылок)
- Метрика нужна сводно через БД-запрос (нельзя ждать первого захода каждого юзера)
- Тяжёлые расчёты (нельзя в auth-флоу)

**Когда lazy лучше:**
- Метрика **личная** для юзера и нужна **именно при его заходе** (стрик, dailyXp)
- Не хочется отдельной инфраструктуры
- Поддержка таймзон важна

В нашем проекте все условия для lazy выполняются.

## Как рассказать на собесе

> «Для дневного XP и стрика я выбрала lazy reset вместо cron. Идея: ничего не делаем по расписанию. При первом запросе юзера за новый день внутри `authenticateRequest` сравниваем "сегодня" с "днём последней активности" в его таймзоне, обнуляем `dailyXp` если нужно, обновляем стрик. Это даёт три плюса: работа делается только для активных юзеров; каждый видит "новый день" в своей tz, а не в UTC; нет отдельной cron-инфраструктуры. Минус — логика сложнее, чем один UPDATE, и нужен throttle, чтобы не пересчитывать при каждом параллельном запросе. Я throttle сделала на час — UI отстаёт максимум на час от факта, юзер этого не замечает.»

## Что важно вынести

- **Lazy reset** — паттерн, где счётчики обнуляются **при следующем взаимодействии**, а не по расписанию.
- **Плюсы**: работа только для активных, поддержка любых таймзон, нет отдельной инфры, нет сбоя cron.
- **Минусы**: сложнее код, нужен throttle.
- **Throttle 1 час** — компромисс между точностью и нагрузкой на БД.
- **Conditional spread** в Prisma `data` — короткая иммутабельная идиома для опциональных полей.
- **Fire-and-forget update** — не блокируем ответ ожиданием БД. Цена: возможна потеря изменений при сбое БД.
- **Мутация локального `user`** — чтобы ответ API содержал свежие значения, пока БД ещё обновляется.
- **TZ через Intl.DateTimeFormat** — без либ.
