# Хуки изнутри: как они работают и почему именно так

`useState`, `useEffect`, `useMemo` — мы их пишем каждый день. Но мало кто может ответить, **почему нельзя вызывать хук внутри `if`**, **где хранится значение между рендерами**, и **почему `setCount(c => c+1)` иногда работает иначе, чем `setCount(count+1)`**. Этот урок — про устройство хуков снизу.

> Главная мысль: хуки — это **функции, которые ищут состояние не у себя, а в текущем активно рендерящемся Fiber-узле**. Связь «хук → его данные» делается через **порядок вызова**, а не имена. Отсюда — все правила хуков.

## Где хранится состояние хука

Когда функциональный компонент рендерится — React выполняет его как обычную функцию. `useState(0)` возвращает `[value, setter]`. Но `value` где-то хранится между рендерами — иначе при следующем рендере он бы сбросился в 0.

Ответ: **внутри Fiber-узла этого компонента**. У каждого Fiber есть поле `memoizedState` — это **связный список хуков**.

Упрощённо:
```js
fiber.memoizedState = {
  memoizedState: 0,         // значение этого хука
  next: {                    // следующий хук
    memoizedState: 'init',
    next: { ... },
  },
};
```

Когда компонент вызывает `useState`, React:
1. Берёт текущий fiber (через глобальный «dispatcher»)
2. Идёт по `memoizedState` к **следующему** хуку в списке
3. Возвращает его значение

Состояние **не имеет имени** — только **позиция в списке**. Первый `useState` — первый узел. Второй — второй. И так далее.

## Почему правила хуков такие жёсткие

Правила хуков:
1. Только на верхнем уровне (не в `if`, не в циклах, не в коллбэках)
2. Только из React-функций (компоненты и кастомные хуки)

Теперь понятно почему:

```js
function Comp() {
  const [a, setA] = useState(0);   // 1-й хук
  if (someCondition) {
    const [b, setB] = useState(0); // 2-й хук — но только иногда!
  }
  const [c, setC] = useState(0);   // 3-й... или 2-й, если if false
  // ...
}
```

При первом рендере `someCondition === true` → хуки в порядке `a, b, c`. При следующем `someCondition === false` → порядок `a, c`. Но React в позиции 2 ожидает state `b`, а получает `c`. **Перепутывание значений**, баг, ад.

Поэтому ESLint-правило `react-hooks/rules-of-hooks` — обязательное. Оно ловит большинство таких ситуаций статически.

## Простая модель useState

Представь себе сильно упрощённую реализацию:

```js
let hooks = [];        // массив хуков текущего компонента
let hookIndex = 0;     // позиция следующего хука

function useState(initial) {
  const i = hookIndex++;
  if (hooks[i] === undefined) {
    hooks[i] = initial;   // первый рендер — инициализация
  }
  const setState = (value) => {
    hooks[i] = typeof value === 'function' ? value(hooks[i]) : value;
    rerender();           // запросить новый рендер
  };
  return [hooks[i], setState];
}

function render(Component) {
  hookIndex = 0;          // на старте рендера — сбрасываем индекс
  return Component();
}
```

Эта модель **точно объясняет**:
- Почему порядок важен (`hooks[i]` берётся по индексу)
- Почему `useState(0)` инициализируется только первый раз (`hooks[i] === undefined`)
- Почему `setState(value)` обновляет `hooks[i]`
- Почему `setState(c => c+1)` принимает функцию (для функциональных апдейтов)

Реальный React сложнее (массив → связный список, dispatcher переключается между mount/update, batching), но **принцип тот же**.

## Stale closure — почему `setCount(c => c+1)` иногда нужен

Помнишь из урока про замыкания: каждый рендер компонента — **новый вызов** функции. Создаются новые локальные переменные, в том числе захваченные хуками.

```js
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);   // ← здесь count из ПЕРВОГО рендера, всегда 0
    }, 1000);
    return () => clearInterval(id);
  }, []);                    // ← пустой массив — эффект запустился один раз

  return <div>{count}</div>;
}
```

Эффект запустился один раз с `count === 0`. `setInterval` создал замыкание над этим `count`. Через секунду: `setCount(0 + 1)` → state становится 1. Через ещё секунду: **снова** `setCount(0 + 1)` (тот же замкнутый ноль). Счётчик никогда не идёт выше 1.

Решений два:

**1. Функциональный апдейт.** `setCount` принимает функцию `prev => newValue`. React передаёт **актуальное** значение state:
```js
setCount(c => c + 1);   // c — всегда текущее значение
```

Сейчас не важно, какой `count` был на момент создания интервала. Функция получает реальный текущий state.

**2. Свежие зависимости.** Передать `count` в массив зависимостей `useEffect` — эффект будет пересоздаваться при изменении count, замыкание всегда свежее. Но `setInterval` тогда **тоже** будет пересоздаваться → мигание, лишняя работа.

Между ними чаще выбирают (1) — функциональный апдейт. Если нужна свежесть других внешних переменных — (2).

## Custom hooks — это просто функции

`useSomething` — обычная функция, которая вызывает другие хуки. Никакой магии:

```js
function useCounter(initial = 0) {
  const [count, setCount] = useState(initial);
  const inc = () => setCount(c => c + 1);
  const dec = () => setCount(c => c - 1);
  return { count, inc, dec };
}
```

«Хуком» функцию делают **два правила**:
1. Префикс `use*` (для ESLint и читаемости)
2. Внутри вызывает другие хуки

React **никак не знает**, что это кастомный хук — для него это обычный вызов функции, в которой случились вызовы хуков. Магии нет.

## useMemo и useCallback — кеш по dependencies

Эти два хука часто переоптимизируют. Их суть:

```js
const value = useMemo(() => expensiveComputation(a, b), [a, b]);
```

- На первый рендер: вычислил, сохранил в `memoizedState` вместе с deps `[a, b]`.
- На следующий: сравнил новые deps с сохранёнными через `Object.is`.
- Если совпадают — вернул кеш.
- Если нет — пересчитал, обновил кеш и deps.

`useCallback` — тот же `useMemo`, но кеширует **функцию** (которую вернёт `() => fn`). По сути `useCallback(fn, deps) === useMemo(() => fn, deps)`.

**Когда они реально нужны:**
1. Вычисление тяжёлое (сортировка большого массива, парсинг)
2. Значение передаётся как пропс в **мемоизированный** компонент (`React.memo`) — иначе тот будет ререндериться от ссылочного неравенства
3. Значение в зависимостях другого хука (`useEffect`), и его пересоздание триггерит лишние эффекты

**Когда не нужны:**
- Простое вычисление: `useMemo(() => a + b, [a, b])` тратит больше, чем экономит
- Локальная функция, не пробрасывается в `memo`: `useCallback` ради красоты — анти-перфоманс

## useRef — хранение без ререндера

```js
const ref = useRef(0);
ref.current = 5;
// компонент НЕ перерендерится
```

`useRef` — хук, который возвращает объект `{ current: ... }`. Меняешь `current` — данные есть, но React **не запускает рендер**. Полезно для:
- Хранения DOM-узла (`<input ref={ref} />`)
- Хранения мутабельных значений между рендерами без UI-апдейта (id таймера, scroll position, флаг «первый рендер»)
- Замены `this` из классов

`useRef` не делает форматирование — это просто `{current: x}` в `memoizedState`. Поэтому он быстрый.

## Dispatcher — почему в одной функции хуки работают, а в другой нет

В React-движке есть **глобальная переменная** `ReactCurrentDispatcher`. Перед каждым рендером компонента React заменяет её на «диспетчер для mount» или «диспетчер для update».

```js
// псевдокод React
function renderWithHooks(Component, fiber) {
  currentFiber = fiber;
  hookIndex = 0;
  ReactCurrentDispatcher.current = fiber.memoizedState === null
    ? mountDispatcher    // первый рендер компонента
    : updateDispatcher;  // повторный
  return Component();
}
```

Сам `useState` это:
```js
function useState(initial) {
  return ReactCurrentDispatcher.current.useState(initial);
}
```

То есть он **просто прокси** к текущему dispatcher. Когда ты вызываешь `useState` **вне компонента** — диспетчер указывает на «invalid», и React бросает `Invalid hook call`.

Это и есть техническая причина правила «хуки только из React-функций» — без активного диспетчера хук не знает, в каком fiber он работает.

## Что важно вынести

- **Хуки хранятся в Fiber-узле** в виде связного списка `memoizedState`. Связь «хук → данные» через **позицию**, не имя.
- **Правила хуков** — следствие позиционной связи. Условный вызов сбивает порядок → перепутывание state.
- **Stale closure** — захват старого state в колбэке. Лечится **функциональным апдейтом** `setX(prev => ...)` или свежими зависимостями.
- **Custom hooks** — обычные функции с префиксом `use*`. React не различает кастомные и встроенные на уровне механики.
- **`useMemo`/`useCallback`** кешируют по deps через `Object.is`. Используй точечно — переоптимизация делает медленнее.
- **`useRef`** хранит данные без ререндера — для DOM-ссылок, мутабельных значений между рендерами, замены `this`.
- **Dispatcher** — глобальная переменная React, которая делает `useState` доступным «изнутри» текущего рендера. Поза неё хуки не работают, отсюда `Invalid hook call`.
