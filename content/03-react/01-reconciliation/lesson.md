# Reconciliation и Fiber: как React решает, что перерисовать

Когда state меняется — React не перерисовывает всё. Он сравнивает «было» и «стало», находит **минимальный набор изменений**, и применяет их к реальному DOM. Этот процесс называется **reconciliation**. С 2017 года он переписан на новую архитектуру — **Fiber**. На собесах мидл+ это спрашивают почти всегда. Этот урок — про устройство.

> Главная мысль: React не работает с реальным DOM напрямую. Он работает с **деревом своих лёгких объектов** (Virtual DOM), сравнивает старое дерево с новым и применяет к DOM только разницу. Это и быстрее, и предсказуемее, чем ручные `appendChild`.

## Virtual DOM в трёх строках

Когда ты пишешь JSX:
```jsx
<div className="card"><span>Привет</span></div>
```

Это компилируется (через Babel) в:
```js
React.createElement('div', { className: 'card' },
  React.createElement('span', null, 'Привет')
);
```

`React.createElement` возвращает **обычный JS-объект**:
```js
{
  type: 'div',
  props: { className: 'card', children: [...] },
  key: null,
  ref: null,
  ...
}
```

Это и есть **vDOM node** — лёгкий описатель того, что должно быть в реальном DOM. Дешёвый в создании, без привязки к реальным DOM-методам.

При следующем рендере создаётся **новое дерево** vDOM. React сравнивает старое и новое, считает diff, применяет к DOM.

## Reconciliation — алгоритм сравнения

Полное сравнение двух деревьев — это **O(n³)** (так в классической CS). Это неприемлемо: React делает рендер на каждое изменение state, нужна скорость.

React использует **эвристики**, которые сводят сложность к **O(n)**:

### Эвристика 1: разные типы → удалить и создать заново

```jsx
// было:
<div>...</div>
// стало:
<span>...</span>
```

React **не пытается** превратить div в span. Удаляет старый узел, создаёт новый. Все state дочерних компонентов теряются (компоненты unmount/mount).

```jsx
// было: <Counter />
// стало: <Display />
```

Разные компоненты → разный тип vDOM-узла → React не пытается «переименовать», а пересоздаёт. Поэтому **меняя тип элемента, ты теряешь его state**.

### Эвристика 2: один тип → diff props

```jsx
<button className="primary" onClick={fn1}>Старый</button>
// →
<button className="primary danger" onClick={fn2}>Новый</button>
```

Тип одинаковый (`button`) → React сохраняет DOM-элемент, только меняет:
- `className` → setAttribute
- `onClick` → переподключает event listener
- text content → обновляет

Это самый частый случай и самая ценная оптимизация.

### Эвристика 3: списки и `key`

Это **частый собесовский вопрос**. Когда у тебя список:
```jsx
{items.map(item => <Row data={item} />)}
```

Если порядок элементов меняется — React не знает, какие элементы переехали, а какие удалились. По умолчанию он сравнивает **по позиции в массиве**. Это часто работает плохо:

```
было:  [Анна] [Борис] [Виктор]
стало: [Виктор] [Анна] [Борис]

React видит:
- позиция 0: Анна → Виктор (мутирует)
- позиция 1: Борис → Анна (мутирует)
- позиция 2: Виктор → Борис (мутирует)
3 мутации вместо «переместить позицию 2 в позицию 0»
```

Это и медленно, и **ломает state** дочерних компонентов: если в `Row` был раскрытый `<input>` с текстом, текст переедет не на ту строку.

Решение — `key`:
```jsx
{items.map(item => <Row key={item.id} data={item} />)}
```

`key` — стабильный идентификатор. React сравнивает узлы **по key**, а не по позиции. Видит «Виктор переехал», физически перемещает DOM-узел, state Row остаётся при нём.

**Правила хороших key:**
- **Стабильные** — не меняются между рендерами
- **Уникальные** в пределах списка (не глобально — внутри `.map`)
- **Не индексы**, если порядок может меняться (`key={i}` — антипаттерн на динамических списках)

`key={item.id}` — почти всегда правильно. `key={Math.random()}` — антипаттерн (каждый рендер новые keys → всё пересоздаётся, state теряется).

## Fiber — новая архитектура (с 2017)

До 2017 reconciliation был **рекурсивным**: React начинал обход дерева и **не мог остановиться**, пока не закончил. Если дерево большое (тысячи узлов), это могло занять **>16мс** — браузер не успевал отрисовать кадр, всё фризилось.

**Fiber** — переписанный reconciler, который умеет **прерываться** и **продолжать**.

### Что такое Fiber-узел

Каждый компонент в дереве представлен `Fiber`-объектом:
```js
{
  type: Component,        // что рендерить
  stateNode: instance,    // ссылка на DOM или экземпляр класса
  child: Fiber | null,    // первый ребёнок
  sibling: Fiber | null,  // следующий брат
  return: Fiber | null,   // родитель
  alternate: Fiber | null,// «версия» из прошлого рендера
  pendingProps: ...,
  memoizedProps: ...,
  memoizedState: ...,
  effectTag: ...,
}
```

Это **связный список** через `child`/`sibling`/`return`, а не классическое дерево с массивом детей. Почему так: при таком формате можно **сохранять позицию обхода** в одном указателе. Прервался — записал «я сейчас на этом fiber'е» — продолжил позже.

### Две фазы

Fiber делает работу в две фазы:

**1. Render phase** (можно прерывать):
- Сравниваются vDOM-деревья
- Создаётся «work-in-progress» дерево
- Считаются изменения, но **DOM не трогается**
- Браузер может прервать, отрисовать кадр, дать слово другим задачам

**2. Commit phase** (синхронная, нельзя прерывать):
- Все накопленные изменения применяются к DOM **за раз**
- Вызываются `useLayoutEffect`, потом `useEffect`

Render phase идёт **через requestIdleCallback / scheduler**. Если за время фрейма (~5мс работы за раз) браузер захочет отрисовать — render phase «отдаёт» управление, продолжает в следующий слот.

### Почему это важно

Старый reconciler:
```
state changed → reconcile → fully sync → DOM → render
                   ↑ блокировка на 50мс на больших деревьях
```

Новый Fiber:
```
state changed → start reconcile → yield → browser renders → continue → ... → commit
                                       ↑ юзер видит плавный UI всё время
```

Это основа всех «новых» фич: `useTransition`, `Suspense` для data fetching, concurrent rendering. Они работают потому, что Fiber умеет **прерывать рендер** и **выкидывать незавершённую работу** при поступлении более приоритетной.

## Concurrent mode и приоритеты

React 18 ввёл **concurrent rendering**. Идея: разные обновления имеют **разные приоритеты**.

- **Urgent** — клики, нажатия клавиш, ввод текста. Нужно отрендерить **немедленно**.
- **Transition** — изменение состояния, не критичное к мс. Например, фильтр большого списка.

```js
import { useTransition } from 'react';

const [isPending, startTransition] = useTransition();

function handleInput(e) {
  setText(e.target.value);              // urgent — сразу
  startTransition(() => {
    setFilteredList(filter(list, e.target.value));  // transition — можно отложить
  });
}
```

Без transition: длинный список пересчитывался **до** того, как буква появлялась в input. С transition: буква появляется сразу, список обновляется когда сможет. Это и есть **прерываемый рендер** Fiber'а в действии.

## Что важно вынести

- **vDOM** — обычные JS-объекты `{type, props, children}`. JSX компилится в `createElement`-вызовы.
- **Reconciliation** = сравнение старого и нового vDOM, применение diff к DOM. Сложность O(n) за счёт эвристик.
- **Эвристики**: разный type → удалить+создать, одинаковый type → diff props, списки → сравнение по key.
- **Key** — стабильный идентификатор. Не индекс, если порядок меняется. Сохраняет state дочерних компонентов при перестановке.
- **Fiber** — новая архитектура с прерываемым рендером. Каждый узел — `Fiber` с `child`/`sibling`/`return` (связный список).
- **Две фазы**: render (прерываемая) и commit (синхронная).
- **Concurrent mode + useTransition** — приоритеты обновлений, отзывчивый UI при тяжёлых пересчётах.
