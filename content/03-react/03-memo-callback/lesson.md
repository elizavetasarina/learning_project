# useMemo и useCallback: когда реально нужно (и когда переоптимизация)

`useMemo` и `useCallback` — самые **переиспользуемые без необходимости** хуки в React. Юниоры заворачивают каждую функцию в `useCallback` «чтобы быстрее», а на собесе мидл+ обязан **ответить, когда не надо**. Этот урок — про три реальных кейса использования и как себя проверять.

> Главная мысль: эти хуки — **не про скорость**, а про **стабильность ссылок** между рендерами. Если стабильная ссылка ничему не нужна, оборачивание ничего не даёт и даже **замедляет** код (хук сам не бесплатный).

## Что делают эти хуки

```ts
const value = useMemo(() => expensiveCalc(a, b), [a, b]);
const fn = useCallback(() => doStuff(a), [a]);
```

**`useMemo`** запоминает **значение**:
- На первом рендере: вычисляет, сохраняет вместе с deps.
- На следующем: сравнивает новые deps с сохранёнными через `Object.is`.
- Если совпали — возвращает кеш. Если нет — пересчитывает.

**`useCallback`** — это **`useMemo` для функций**:
```ts
useCallback(fn, deps) === useMemo(() => fn, deps)
```

То есть кеширует **саму функцию**. Возврат — стабильная ссылка, пока deps не изменились.

## Сначала — что НЕ дают эти хуки

### Не делают «компонент быстрее»

Распространённый миф:
> «Я обернул функцию в useCallback, теперь компонент рендерится быстрее»

**Нет.** Компонент всё равно рендерится. `useCallback` только гарантирует, что **ссылка на функцию** не меняется между рендерами. На скорость самого render — **никак** не влияет.

### Не «предотвращают вызов функции»

```ts
const handleClick = useCallback(() => {
  console.log('click');
}, []);
```

Это **не значит** «функция выполнится один раз». Она выполнится каждый раз при клике. `useCallback` влияет только на **ссылку**, не на поведение.

### Не бесплатные

Каждый `useMemo` / `useCallback`:
- Берёт слот в `memoizedState` Fiber'а (~50 байт памяти)
- Сравнивает deps через `Object.is` (для каждого элемента)
- Хранит замыкание над предыдущим значением

Для **простой арифметики** это часто **дороже**, чем посчитать заново:
```ts
const sum = useMemo(() => a + b, [a, b]);   // ← перебор
const sum = a + b;                            // ← так лучше
```

## Три реальных кейса

Полезно использовать только тогда, когда совпадает один из трёх случаев:

### Кейс 1: дорогое вычисление

```ts
const sortedAndFiltered = useMemo(() => {
  return items
    .filter(complexPredicate)
    .sort(complexSort);
}, [items]);
```

Сортировка большого массива, парсинг JSON, recursive walk дерева — это **реально дорого**. `useMemo` спасает: не пересчитываем при каждом рендере, если `items` не изменился.

**Как понять «дорого»:** профайлер. Если в Chrome DevTools Performance ты видишь, что вычисление занимает **> 1ms** и происходит чаще, чем меняются deps — оборачивай.

### Кейс 2: значение пропускаем в `React.memo` компонент

```tsx
const MemoizedChild = React.memo(function Child({ data }) {
  // тяжёлый рендер
});

function Parent() {
  const data = { id: 1, name: 'X' };  // ← новый объект каждый рендер
  return <MemoizedChild data={data} />;
}
```

`React.memo` пропустит ре-рендер, **если пропсы не изменились по ссылке**. Но `data` — новый объект каждый рендер → ссылка разная → `memo` бесполезен.

**Фикс:**
```tsx
function Parent() {
  const data = useMemo(() => ({ id: 1, name: 'X' }), []);
  return <MemoizedChild data={data} />;
}
```

Теперь ссылка `data` стабильна → `React.memo` действительно скипает рендер. То же для функций — `useCallback` нужен для функций, передаваемых в `memo`-компоненты.

### Кейс 3: значение в зависимостях другого хука

```tsx
useEffect(() => {
  doSomethingWith(config);
}, [config]);
```

Если `config` — объект, который пересоздаётся каждый рендер, **`useEffect` будет запускаться на каждом рендере** (даже если содержимое не изменилось). Это **бесконечный цикл** в худшем случае.

**Фикс:**
```tsx
const config = useMemo(() => ({ url, timeout: 5000 }), [url]);
useEffect(() => {
  doSomethingWith(config);
}, [config]);
```

Теперь `config` стабилен пока `url` тот же → `useEffect` запускается только при изменении `url`.

То же для функций, которые в зависимостях другого хука:
```tsx
const handler = useCallback(() => fn(value), [value]);
useEffect(() => {
  ws.on('message', handler);
  return () => ws.off('message', handler);
}, [handler]);
```

## Антипаттерны

### Антипаттерн 1: `useCallback` ради «оптимизации»

```tsx
function Form() {
  const handleSubmit = useCallback(() => {
    submit();
  }, []);

  return <button onClick={handleSubmit}>Submit</button>;
}
```

`<button>` — обычный HTML, не `React.memo`. Ему **всё равно**, изменилась ли ссылка. `useCallback` тут — **чистый шум**: лишняя работа на каждом рендере, ничего не оптимизирует. Должно быть:

```tsx
function Form() {
  function handleSubmit() {
    submit();
  }
  return <button onClick={handleSubmit}>Submit</button>;
}
```

### Антипаттерн 2: `useMemo` для простых вычислений

```tsx
const fullName = useMemo(() => `${first} ${last}`, [first, last]);
```

Конкатенация строк занимает **доли наносекунды**. `useMemo` сам — десятки наносекунд (сравнение deps, доступ к memo store). Чистый минус.

```tsx
const fullName = `${first} ${last}`;  // лучше
```

### Антипаттерн 3: deps массив с пропуском

```tsx
const fn = useCallback(() => {
  doSomething(value);  // ← value не в deps
}, []);
```

Stale closure. `value` захвачен по первому рендеру, навсегда. ESLint правило `react-hooks/exhaustive-deps` это ловит — **не игнорируй его** без понимания.

### Антипаттерн 4: useMemo вокруг React.memo

```tsx
const MemoChild = useMemo(() => React.memo(Child), []);
```

`React.memo` уже мемоизирует. Оборачивать его в `useMemo` бессмысленно. Если речь о условном рендере — это **очень редко** нужно.

## Как себя проверять

Чек-лист перед тем как обернуть в `useMemo`/`useCallback`:

1. **Это передаётся в `React.memo`-компонент?**
   - Да → может быть оправдано
   - Нет → почти наверняка не нужно

2. **Это в зависимостях другого хука?**
   - Да → может быть оправдано
   - Нет → почти наверняка не нужно

3. **Вычисление реально дорогое (> 1мс)?**
   - Да → оправдано
   - Нет → почти наверняка не нужно

Если ни одно «да» — **не оборачивай**.

## Про React Compiler

В React 19+ постепенно роллится **React Compiler** — он автоматически добавляет мемоизацию там, где она нужна. Идея: писать «как удобно», компилятор разберётся.

На сегодня (середина 2025): экспериментальная поддержка. На собесе про это знать **полезно** — показывает, что следишь за развитием React.

Если Compiler станет дефолтом — `useMemo`/`useCallback` в коде станут **редкими** в новом коде. Но в существующих проектах их полно — знать механику обязательно.

## Особый случай: контекст

Если значение `value` в `<Context.Provider value={value}>` — новый объект каждый рендер, все подписчики контекста ре-рендерятся, даже если содержимое не изменилось:

```tsx
function App() {
  const value = { user, theme };  // новый объект каждый рендер
  return <Ctx.Provider value={value}>...</Ctx.Provider>;
}
```

**Фикс:**
```tsx
const value = useMemo(() => ({ user, theme }), [user, theme]);
```

Это **классический** случай, где `useMemo` обязателен. На собесе любят спросить «почему все мои контекст-потребители ре-рендерятся даже когда я ничего не менял?» — ответ про ссылочное равенство.

## Что важно вынести

- **`useMemo`** = кеш значения по deps. **`useCallback`** = `useMemo` для функций.
- Они **не делают код быстрее по умолчанию** — только **сохраняют ссылки**.
- Каждый хук стоит ~десятки наносекунд + слот памяти. Для простых вычислений — **дороже** чем без него.
- **Три реальных кейса**: дорогое вычисление, проп в `React.memo`, deps другого хука.
- **Антипаттерны**: оборачивание для «оптимизации» без причины, простая арифметика, пропуск deps, обёртка вокруг `React.memo`.
- **Context-значение** — почти всегда `useMemo`, иначе все потребители рендерятся.
- **React Compiler** в 19+ может убить нужду в этих хуках — следи за развитием.
