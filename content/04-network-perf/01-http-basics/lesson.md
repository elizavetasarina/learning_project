# HTTP basics: методы, статусы, заголовки

HTTP — протокол, на котором стоит весь веб. Frontend-разработчик дёргает его каждую секунду. На собесе всегда есть блок «что такое REST, что такое 401, зачем PUT отличается от POST». Без этого мидл не закроешь. Этот урок — про структуру.

> Главная мысль: HTTP — это **текстовый протокол запрос/ответ** поверх TCP. Каждый запрос состоит из трёх частей: первая строка (метод + путь), заголовки и тело. Ответ так же: первая строка (статус), заголовки, тело.

## Анатомия запроса

Простой `fetch('/api/users')` с JS превращается в:

```
GET /api/users HTTP/1.1
Host: example.com
Accept: application/json
Cookie: session=abc123
User-Agent: Mozilla/5.0

(пустое тело)
```

Что внутри:
- **Метод + путь + версия** — `GET /api/users HTTP/1.1`
- **Заголовки** — пары `Имя: Значение`. Метаданные о запросе
- **Пустая строка** — разделитель
- **Тело** (для POST/PUT) — данные

И ответ:
```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=60

[{"id":1,"name":"Лиза"}]
```

- **Статус** — `200 OK`
- **Заголовки**
- **Тело** — обычно JSON, HTML, или бинарные данные

## Главные методы

| Метод | Зачем | Идемпотентен? | Тело? |
|---|---|---|---|
| **GET** | Получить ресурс | Да | Нет |
| **POST** | Создать или «сделать что-то» | Нет | Да |
| **PUT** | Полностью заменить ресурс | Да | Да |
| **PATCH** | Частично обновить | Не обязательно | Да |
| **DELETE** | Удалить ресурс | Да | Опционально |
| **HEAD** | Как GET, но только заголовки (без тела) | Да | Нет |
| **OPTIONS** | Узнать что разрешено (CORS preflight) | Да | Нет |

### Что такое идемпотентность

**Идемпотентная операция: повторное выполнение не меняет результат после первого.**

- `GET /users/1` — сколько раз ни запросишь, получишь то же
- `PUT /users/1 {name: 'L'}` — после первого вызова состояние = {name: 'L'}; после второго — то же
- `DELETE /users/1` — после первого удалили, после второго ничего нет (или 404)
- `POST /users` — **не идемпотентна!** Каждый раз создаёт нового юзера

Это важно для **retry-логики**: можно безопасно повторять PUT/DELETE при сетевом сбое, а POST — нельзя (риск дубля).

PATCH **может** быть идемпотентным или нет — зависит от семантики. `PATCH /users/1 {age: 30}` — идемпотентно. `PATCH /counter/1 {action: 'increment'}` — нет.

## Статусы (коды состояния)

**Группы по первой цифре:**

| Группа | Что значит |
|---|---|
| **1xx** | Информационные (редко встречаются) |
| **2xx** | Успех |
| **3xx** | Редирект |
| **4xx** | Ошибка клиента |
| **5xx** | Ошибка сервера |

### Самые важные

**2xx — успех:**
- **200 OK** — обычный успех с телом ответа
- **201 Created** — создано (для POST после успешного создания)
- **204 No Content** — успех без тела (для DELETE, или PUT без возврата)

**3xx — редиректы:**
- **301 Moved Permanently** — навсегда переехало, поисковики обновляют ссылки
- **302 Found** — временный редирект
- **304 Not Modified** — ничего не изменилось с прошлого раза (в связке с кэшем)

**4xx — клиент виноват:**
- **400 Bad Request** — неправильный запрос (битый JSON, невалидные данные)
- **401 Unauthorized** — не аутентифицирован (нет токена или невалидный)
- **403 Forbidden** — аутентифицирован, но нет прав на это действие
- **404 Not Found** — ресурс не найден
- **409 Conflict** — конфликт (например, email уже занят)
- **422 Unprocessable Entity** — синтаксически OK, но семантически нет (валидация)
- **429 Too Many Requests** — rate limit

**5xx — сервер виноват:**
- **500 Internal Server Error** — что-то упало
- **502 Bad Gateway** — сервер-прокси не дождался от бэка
- **503 Service Unavailable** — временно недоступно (можно retry)
- **504 Gateway Timeout** — таймаут

### Часто путают

- **401 vs 403** — 401 «не знаю, кто ты», 403 «знаю, но нельзя»
- **404 vs 410** — 404 «нет сейчас», 410 «было, но удалено навсегда»
- **400 vs 422** — 400 «не понял запрос», 422 «понял, но данные кривые»

## Заголовки

Заголовков много, базовые знать обязательно.

### Запрос

- **`Accept: application/json`** — какой формат ответа предпочитает клиент
- **`Content-Type: application/json`** — формат тела запроса
- **`Authorization: Bearer <token>`** — токен (Bearer, Basic, и т.д.)
- **`Cookie: session=abc`** — куки сессии
- **`User-Agent: ...`** — что за клиент
- **`If-None-Match: "abc123"`** — для кэширования (ETag)
- **`If-Modified-Since: ...`** — для кэширования по дате
- **`Origin: https://app.com`** — для CORS

### Ответ

- **`Content-Type: application/json`** — формат тела
- **`Content-Length: 1234`** — размер тела в байтах
- **`Set-Cookie: session=abc; HttpOnly; Secure`** — установить куки
- **`Cache-Control: max-age=3600`** — кэширование (об этом отдельный урок)
- **`ETag: "abc123"`** — версия ресурса для кэша
- **`Location: /users/5`** — куда редирект или какой ресурс создан
- **`Access-Control-Allow-Origin: *`** — CORS

## REST на пальцах

REST (Representational State Transfer) — это **набор принципов**, не строгий стандарт. Главные:

1. **Stateless** — сервер не хранит сессию клиента между запросами. Каждый запрос содержит всё нужное (например, токен).
2. **Ресурсо-ориентированный URL** — `/users/1`, не `/getUser?id=1`. URL описывает **что**, не «как».
3. **Методы передают семантику** — `GET /users/1` (читать), `DELETE /users/1` (удалить), `PUT /users/1` (заменить).
4. **Множественное число** — `/users` (коллекция), `/users/1` (один).

### Классическая REST-таблица

| Метод | URL | Что делает |
|---|---|---|
| GET | `/users` | Список юзеров |
| GET | `/users/1` | Один юзер |
| POST | `/users` | Создать юзера |
| PUT | `/users/1` | Заменить юзера целиком |
| PATCH | `/users/1` | Частично обновить |
| DELETE | `/users/1` | Удалить |

В реальном проекте редко чисто-REST: бывают `/users/1/activate`, `/auth/login` — это RPC-стиль. Это нормально, главное чтобы было консистентно.

### Альтернативы REST

- **GraphQL** — один эндпоинт, клиент указывает в теле что нужно. Удобно для сложных вложенных данных.
- **gRPC** — бинарный протокол поверх HTTP/2. Быстро, используется в межсервисном общении.
- **tRPC** — типобезопасный RPC для TypeScript. Сервер и клиент шарят типы.

## Версии HTTP — коротко

- **HTTP/1.0** — каждое соединение одноразовое
- **HTTP/1.1** — keep-alive (переиспользование соединения), но запросы по одному (head-of-line blocking)
- **HTTP/2** — multiplexing (несколько запросов параллельно через одно соединение), бинарный, server push
- **HTTP/3** — поверх UDP (QUIC), решает head-of-line blocking на сетевом уровне

Сегодня большинство сайтов уже на HTTP/2. HTTP/3 распространяется. Знать различия — собесовый плюс.

## Что важно вынести

- **HTTP** — текстовый протокол: первая строка (метод/статус) + заголовки + пустая строка + тело.
- **Методы**: GET (читать), POST (создать/действие), PUT (заменить), PATCH (обновить), DELETE.
- **Идемпотентность** — повтор не меняет результат. Идемпотентны: GET, PUT, DELETE. Не идемпотентен POST.
- **Статусы по группам**: 2xx OK, 3xx редирект, 4xx клиент, 5xx сервер.
- **Ключевые коды**: 200, 201, 204, 301, 304, 400, 401, 403, 404, 409, 422, 429, 500, 502, 503.
- **401 ≠ 403**: 401 «кто ты?», 403 «нельзя».
- **Базовые заголовки**: Accept, Content-Type, Authorization, Cookie, Cache-Control, ETag, Origin.
- **REST**: stateless, ресурсо-ориентированные URL, методы передают семантику. Не догма.
- **HTTP/2 + multiplexing** — главное отличие от HTTP/1.1.
