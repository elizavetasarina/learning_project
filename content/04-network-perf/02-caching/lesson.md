# HTTP кэширование: Cache-Control, ETag, SWR

«Самый быстрый запрос — тот, который не был сделан». HTTP-кэш экономит трафик, ускоряет загрузку, разгружает сервер. На собесе по сети и performance обязательно спросят: про `Cache-Control` и его директивы, про `ETag`, про разницу с `Last-Modified`, и про современный `stale-while-revalidate`. Этот урок — про механизмы.

> Главная мысль: кэширование — это **договор** между сервером и клиентом, передаваемый через **HTTP-заголовки**. Сервер говорит «эти данные валидны столько-то секунд» или «вот идентификатор версии». Клиент решает, можно ли использовать локальную копию.

## Зачем

Без кэша каждое открытие сайта — это десятки HTTP-запросов: HTML, CSS, JS, шрифты, картинки, API. На медленной сети это легко 3-5 секунд. С правильным кэшем второе открытие — **миллисекунды**: всё берётся локально.

Кэш живёт **на нескольких уровнях**:
- **Браузер** (диск + память)
- **CDN** (Cloudflare, Vercel Edge, AWS CloudFront)
- **Reverse proxy** (nginx, Varnish)
- **Сервис-воркер** (для PWA)

Все эти уровни используют **одни и те же HTTP-заголовки**. Понимать кэш = понимать эти заголовки.

## Главный заголовок: `Cache-Control`

```
Cache-Control: public, max-age=3600
```

Это **набор директив** через запятую. Самые важные:

| Директива | Смысл |
|---|---|
| `max-age=N` | Свежий в течение N секунд. Самая важная |
| `public` | Можно кэшировать **где угодно** (в т.ч. CDN, прокси) |
| `private` | Только в **личном** кэше (браузер юзера). Не CDN |
| `no-cache` | Можно хранить, но **проверять** перед использованием (conditional request) |
| `no-store` | Не хранить **вообще нигде**. Для приватных/чувствительных данных |
| `immutable` | Не изменится до истечения max-age, не делать conditional request |
| `must-revalidate` | По истечении max-age — **обязательно** запросить заново, не возвращать stale |
| `stale-while-revalidate=N` | Можно вернуть stale ещё N секунд, **параллельно** обновляя в фоне |

### `no-cache` vs `no-store` — частая путаница

- **`no-cache`** = «можно хранить, но **спрашивай** перед каждым использованием». Браузер делает conditional request — сервер отвечает либо `200` (новые данные), либо `304 Not Modified` (используй кэш).
- **`no-store`** = «не сохраняй вообще». Каждое использование = полный запрос.

`no-cache` ≠ «никогда не кэшировать». Это **«всегда проверять»**.

## ETag — отпечаток версии

Сервер может прислать `ETag`:
```
ETag: "abc123"
```

Это **идентификатор версии ресурса**. При следующем запросе клиент шлёт:
```
If-None-Match: "abc123"
```

Сервер сравнивает с текущим ETag:
- Совпадает → `304 Not Modified`, тело не отправляется. Клиент использует свой кэш.
- Не совпадает → `200 OK` с новым телом и новым ETag.

**Экономия:** при 304 трафика передаётся **меньше килобайта** (только заголовки), вместо мегабайт ресурса.

Как сервер генерирует ETag:
- Хэш содержимого (`md5(body)`)
- Версия из БД
- Timestamp модификации
- Хэш версии деплоя (для статики)

Главное — **уникальный для версии**.

## `Last-Modified` — альтернатива ETag

```
Last-Modified: Wed, 09 Jun 2026 12:00:00 GMT
```

Сервер шлёт **дату последнего изменения**. Клиент:
```
If-Modified-Since: Wed, 09 Jun 2026 12:00:00 GMT
```

Сервер проверяет, изменился ли ресурс после этой даты. Если нет → `304`. Если да → `200` с новой версией.

### ETag vs Last-Modified

| | ETag | Last-Modified |
|---|---|---|
| Точность | Любая (хэш контента) | До секунды |
| Поддержка частых изменений | Да (хэш меняется при любом изменении) | Слабая (за 1 секунду может быть несколько правок) |
| Для динамики | Да | Нет (сервер не знает «дату» вычисленного ответа) |
| Для статики | Да | Да |

**Рекомендация:** для статики — оба норм. Для динамики — `ETag`. Если хэшируешь — это **strong ETag**. Если что-то более расплывчатое — **weak ETag** (`W/"abc"`).

## Conditional requests — как это работает

Слева браузер, справа сервер:

```
[1] Первый запрос:
    Browser → GET /style.css
    Server → 200 OK
             Cache-Control: max-age=3600
             ETag: "v1"
             [тело css]

[2] Через минуту (в пределах max-age):
    Browser → использует кэш, БЕЗ запроса

[3] Через 2 часа (max-age истёк):
    Browser → GET /style.css
              If-None-Match: "v1"
    Server → 304 Not Modified (или 200 с новой версией)

[4] При 304:
    Browser → продлевает кэш, использует локальный файл
```

Это и есть жизненный цикл кэшированного ресурса.

## `stale-while-revalidate` (SWR)

Современный паттерн:
```
Cache-Control: max-age=10, stale-while-revalidate=86400
```

Что значит:
- В течение **10 секунд** — данные свежие, отдаются мгновенно
- Между 10 секунд и 86400 (сутки) — данные **stale, но можно использовать**. Браузер отдаёт **сразу из кэша** + **параллельно** обновляет в фоне
- После суток — обязательный полный запрос

**Зачем:** дать юзеру **мгновенный** ответ из кэша, при этом не показывать действительно устаревшие данные. Это **идеально** для «полу-динамики»: списки, фиды, превью.

Имя популярного React-хука **SWR** от Vercel — отсылка к этому заголовку.

## Кэш браузера: уровни

Браузер хранит кэш в двух местах:

1. **Memory cache** — в RAM текущей вкладки. Быстрый, но погибает при закрытии.
2. **Disk cache** — на диске. Переживает перезагрузки.

В DevTools → Network колонка `Size`:
- `(memory cache)` — из RAM
- `(disk cache)` — с диска
- Число (например `42 kB`) — реальный сетевой запрос

Hard reload (Ctrl+Shift+R) **игнорирует** кэш — браузер шлёт `Cache-Control: no-cache` сам.

## Когда что использовать

### Статика с хэшем в имени файла

`/static/app.abc123.js` — содержимое **никогда не меняется** (новая версия = новое имя).

```
Cache-Control: public, max-age=31536000, immutable
```

Год хранения, не делать conditional request — мы **гарантируем** что не изменится. Это **самый агрессивный** кэш, для всего что хэшируется при билде.

### HTML страницы

```
Cache-Control: no-cache
```

Каждый раз проверять — HTML маленький, но может содержать ссылки на новую статику. Проверять обязательно.

### Динамика — список постов

```
Cache-Control: private, max-age=10, stale-while-revalidate=60
```

10 секунд мгновенно, минуту с фоновым обновлением. Юзеру кажется супер-быстро.

### Личный API — профиль юзера

```
Cache-Control: private, max-age=0, must-revalidate
```

Не кэшируем (max-age=0). Можно положить в браузер, но **обязательно** проверять. Или просто `no-store`.

### Чувствительные данные — токен сессии, банковский баланс

```
Cache-Control: no-store
```

Не хранить нигде. Только текущий ответ, потом — забыть.

## Cache invalidation

«Инвалидация кэша — одна из двух сложных задач в CS» (Фил Карлтон).

Стратегии:

**1. Версионирование URL**
```
/posts.js?v=2024-06-09
```
Новая версия = новый URL → старый кэш не задействуется.

**2. Хэш в имени файла**
```
/posts.abc123.js → /posts.def456.js
```
То же, но для статики. Стандарт у webpack/Vite/Next.

**3. Cache busting через ETag**
Сервер меняет ETag при обновлении контента. Conditional request получит `200` с новой версией.

**4. Purge на CDN**
Cloudflare/Vercel/CloudFront дают API: «удали этот URL из кэша». После деплоя — purge.

**5. `revalidatePath` / `revalidateTag` в Next.js**
Серверная сторона: после мутации обнули кэш для пути или тега.

## В нашем проекте

Сейчас у нас **минимальное** кэширование:

- API возвращают `force-dynamic` (Next.js): кэш отключён
- Статика обслуживается Vercel: автоматически с хэшем в имени + immutable

**Что можно улучшить:**
- `GET /api/reviews/due` мог бы иметь `Cache-Control: private, max-age=60` — список к повтору не меняется чаще раза в минуту
- `GET /api/progress` тот же кейс

Но в нашем проекте `force-dynamic` намеренно — данные часто меняются после ответов, оптимизация преждевременна.

## Что важно вынести

- **`Cache-Control`** — главный заголовок. Самые важные директивы: `max-age`, `public`/`private`, `no-cache`, `no-store`, `immutable`, `stale-while-revalidate`.
- **`no-cache`** ≠ «не кэшировать». Это «всегда проверять» (conditional request).
- **`ETag`** = отпечаток версии. **`If-None-Match`** в запросе → `304 Not Modified` если совпало.
- **`Last-Modified`** + **`If-Modified-Since`** = старый аналог. Менее точно (до секунды), но проще.
- **`stale-while-revalidate`** — современный паттерн: мгновенный stale + фоновое обновление.
- **Иерархия кэша**: браузер (RAM + диск) → CDN → reverse proxy → сервер. Один заголовок — все уровни.
- **Хэш в имени файла** + `immutable` — золотой стандарт для статики.
- **Инвалидация**: версии в URL, хэши в именах, CDN purge, `revalidatePath`/`Tag` в Next.js.
