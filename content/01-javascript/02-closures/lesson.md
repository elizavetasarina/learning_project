# Замыкания и область видимости

Замыкание (closure) — это функция вместе с её лексическим окружением: переменными, которые были доступны в месте, где функция была **создана**. Это одна из самых частых тем на собесах и одна из тех, где легко запутаться, если не понять механику.

> Ключевая идея: функция «запоминает» переменные из места своего рождения, а не из места своего вызова. Куда бы функцию потом ни передали — она тащит за собой это окружение.

## Сначала — область видимости (scope)

Чтобы понять замыкания, нужно понять scope. Область видимости — это правила, определяющие, к каким переменным код имеет доступ. В JavaScript scope лексический: он определяется тем, **где** в коде написана функция, а не откуда её вызвали.

Когда код ищет переменную, он смотрит сначала в своей области, потом во внешней, потом ещё выше — до глобальной. Это называется цепочкой областей видимости (scope chain).

Внутренняя функция видит всё, что снаружи неё: свои переменные, переменные внешней функции, глобальные. А вот наоборот — нет: внешний код не видит внутренние переменные. Доступ идёт только изнутри наружу.

## Что такое замыкание

Замыкание возникает, когда внутренняя функция использует переменные внешней — и эту внутреннюю функцию «выносят» наружу (возвращают, передают в коллбэк, сохраняют). Внешняя функция уже завершилась, но её переменные не исчезли — потому что внутренняя функция их держит.

```js
function makeCounter() {
  let count = 0;            // переменная внешней функции
  return function () {
    count++;                // внутренняя функция помнит count
    return count;
  };
}

const counter = makeCounter();
counter(); // 1
counter(); // 2
counter(); // 3
```

`makeCounter()` уже завершилась после первой строки. По обычной логике `count` должна была исчезнуть. Но возвращённая функция замкнулась на `count` — и переменная живёт ровно столько, сколько живёт эта функция.

## Зачем это нужно на практике

Замыкания — не академическая абстракция, ты используешь их постоянно, даже не замечая:

Приватные данные (инкапсуляция) — переменная `count` недоступна снаружи, её можно менять только через возвращённую функцию. Это приватность без классов.

Колбэки и обработчики событий — когда ты пишешь обработчик внутри функции, он замыкается на переменные этой функции:

```js
function setup(name) {
  button.addEventListener("click", () => {
    console.log("Привет, " + name); // помнит name навсегда
  });
}
```

Хуки в React — `useState` устроен на замыканиях. Когда ты вызываешь `setCount`, он работает с переменной, на которую замкнулся при создании компонента.

## Классическая ловушка собеседований

Самый частый каверзный вопрос — цикл с `var`:

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// выведет: 3, 3, 3 — не 0, 1, 2!
```

Все три коллбэка замкнулись на **одну и ту же** переменную `i` (потому что `var` имеет функциональную область, не блочную). К моменту срабатывания таймеров цикл закончился, и `i` уже равна 3.

Решение — `let`, который создаёт новую переменную на каждой итерации:

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// выведет: 0, 1, 2 — каждый коллбэк замкнулся на свою i
```

> Это любимый вопрос интервьюеров, потому что он проверяет сразу две вещи: понимание замыканий и разницу между var и let по области видимости.

## Что важно вынести

Область видимости в JavaScript лексическая — определяется местом написания кода, не местом вызова. Замыкание — это функция плюс окружение, которое она запомнила при создании. Переменные внешней функции не удаляются, пока на них держит ссылку внутренняя функция, даже если внешняя уже завершилась. Замыкания дают приватность, питают колбэки и обработчики, и лежат в основе React-хуков. Классическая ловушка с `var` в цикле решается переходом на `let`.
