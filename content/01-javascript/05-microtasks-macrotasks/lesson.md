# Микро- и макрозадачи: event loop без воды

JavaScript однопоточный. У него **один call stack**. Но при этом ты можешь делать `fetch`, `setTimeout`, `Promise.then` — и они «работают параллельно». Как? Через **event loop** и две очереди задач. Этот урок — про то, что движок реально делает между «вызвал `setTimeout`» и «колбэк выполнился».

> Ключевая мысль: event loop — это не «фоновый поток JS». Это **механизм координации** между стеком и очередями. Сам JS-код всё ещё выполняется в один поток. Параллелизм даёт окружение (браузер, Node), а не язык.

## Стек: что выполняется прямо сейчас

**Call stack** — стек вызовов функций. Когда ты вызываешь `f()`, на стек кладётся frame `f`. Внутри `f` вызывается `g()` — на стек кладётся `g`. `g` завершается — снимается. `f` завершается — снимается.

```js
function a() { b(); }
function b() { c(); }
function c() { console.log("hi"); }
a();
// Stack во время console.log: [a, b, c, console.log]
```

**Принципиально:** пока стек не пуст, ничего другого выполняться не может. Это и означает «однопоточный JS».

```js
while (true) {} // повесит вкладку — стек никогда не очищается
```

## Web API: что не в JS-движке

`setTimeout`, `fetch`, обработчики DOM-событий — это **не JavaScript**. Их предоставляет окружение (браузер через Web API, Node.js через libuv).

Когда ты пишешь:
```js
setTimeout(callback, 1000);
```

Движок не «ждёт» 1000мс. Он передаёт `(callback, 1000)` в Web API и идёт дальше. Web API запускает таймер в фоне (на другом потоке, в системе ОС). Когда время вышло — кладёт `callback` в очередь.

Это объясняет, почему `setTimeout(f, 0)` всё равно не выполнится мгновенно: он встаёт в очередь, а очередь обрабатывается только когда стек пуст.

## Две очереди: микро и макро

Тут начинается интересное. Очередь не одна — их **две** (грубо), с разным приоритетом.

**Очередь макротасок** (macrotask queue, иногда просто task queue):
- `setTimeout`, `setInterval`
- I/O (в Node)
- Обработчики DOM-событий (click, scroll, ...)
- `MessageChannel.postMessage`
- Рендеринг (в браузере)

**Очередь микротасок** (microtask queue):
- `Promise.then` / `.catch` / `.finally` колбэки
- `queueMicrotask(fn)`
- `MutationObserver`
- `await` (это сахар над `then`)

## Главное правило: микротаски всегда раньше

Алгоритм event loop:
```
loop:
  1. Взять одну макротаску из очереди → выполнить (стек заполняется и опустошается)
  2. Выполнить ВСЕ микротаски (пока очередь не пуста)
  3. (В браузере) Перерисовать DOM при необходимости
  4. Идти на 1
```

Принципиальное: **между двумя макротасками всегда полностью опустошается очередь микротасок**. Даже если микротаска поставила новую микротаску — она тоже выполнится до следующей макротаски.

Это значит:
- `Promise.then` выполнится **раньше**, чем `setTimeout(f, 0)`, даже если `setTimeout` поставлен раньше.
- Между `setTimeout`-колбэком и следующим колбэком есть «пауза» для всех микротасок.

## Классическая задача

```js
console.log("1");
setTimeout(() => console.log("2"), 0);
Promise.resolve().then(() => console.log("3"));
console.log("4");
```

Порядок: **1, 4, 3, 2**.

Разбор:
1. `console.log("1")` — синхронно, стек.
2. `setTimeout(...)` — колбэк уходит в Web API → таймер → очередь макротасок.
3. `Promise.resolve().then(...)` — колбэк сразу в очередь микротасок.
4. `console.log("4")` — синхронно, стек.
5. Стек опустел. Event loop:
   - Микротаски: выполнить `console.log("3")`.
   - Затем макротаска: `console.log("2")`.

Эту задачу спрашивают **постоянно**. Уметь объяснять через очереди — обязательно.

## Усложнённая версия

```js
console.log("A");

setTimeout(() => console.log("B"), 0);

Promise.resolve().then(() => {
  console.log("C");
  Promise.resolve().then(() => console.log("D"));
});

Promise.resolve().then(() => console.log("E"));

console.log("F");
```

Порядок: **A, F, C, E, D, B**.

Разбор:
1. Синхронно: `A`, `F`.
2. После стека — выполняем **все** микротаски:
   - `C` (из первого `then`).
   - `C` поставила ещё одну микротаску (`D`) — но сначала остальные уже стоящие в очереди.
   - `E` (из второго `then`).
   - Теперь очередь содержит `D` — выполняем.
3. Только теперь — макротаска: `B`.

Главный нюанс: микротаска, поставленная внутри другой микротаски, выполнится **до перехода к макротасках**. Если бы в `C` стояла бесконечная цепочка `.then().then()...` — `setTimeout("B")` никогда бы не выполнился (microtask starvation).

## `await` — это `.then` под маской

```js
async function foo() {
  console.log("1");
  await Promise.resolve();
  console.log("2");
}
foo();
console.log("3");
```

Порядок: **1, 3, 2**.

Почему: `await` — синтаксический сахар. Этот код эквивалентен:
```js
function foo() {
  console.log("1");
  return Promise.resolve().then(() => {
    console.log("2");
  });
}
```

То есть всё после `await` — это `.then`-колбэк, микротаска. Поэтому `3` выводится раньше `2`.

## Старая ловушка про вложенные таймауты

В вопросах для мидлов любят:

```js
setTimeout(() => {
  console.log("outer");
  Promise.resolve().then(() => console.log("inner"));
}, 0);

setTimeout(() => console.log("next"), 0);
```

Порядок: **outer, inner, next**.

Почему: после `outer` стек пуст → event loop опустошает микротаски → `inner`. Только потом следующая макротаска → `next`.

Если бы микротаски не опустошались между макротасками — было бы `outer, next, inner`. Эта разница — и есть смысл «микротаски всегда раньше».

## queueMicrotask и зачем он нужен

Если тебе нужно отложить выполнение «после текущего синхронного кода, но до любой макротаски» — используй `queueMicrotask`:

```js
queueMicrotask(() => console.log("micro"));
console.log("sync");
// "sync", "micro"
```

Эквивалент `Promise.resolve().then(fn)`, но без создания промиса. Используется в либах (например, для «defer to next tick» без перфоманс-овэрхеда промиса).

В Node есть `process.nextTick` — ещё более ранняя очередь, специфичная для Node. На фронте её нет.

## Рендеринг в браузере: где он встраивается

Между макротасками браузер вставляет **рендер-фазу** (когда нужно — обычно ~60 раз в секунду):
- Style recalc
- Layout
- Paint
- Composite

Поэтому если ты делаешь много синхронной работы в одной макротаске — браузер не успевает рендерить, страница «фризит». Решение — разбивать на куски через `setTimeout(0)` или `requestIdleCallback`:

```js
function processChunk(arr, i = 0) {
  if (i >= arr.length) return;
  doWork(arr[i]);
  setTimeout(() => processChunk(arr, i + 1), 0); // даём браузеру отрендерить
}
```

**`requestAnimationFrame`** — особая очередь, синхронизирована с частотой кадров. Колбэк вызывается перед следующим рендером. Используется для анимаций.

## Node.js: своя специфика

Event loop в Node устроен похоже, но с дополнительными фазами (timers, pending callbacks, poll, check, close). И ещё есть `process.nextTick` (приоритетнее микротасок) и `setImmediate` (свой тип макротаски).

Для собеса фронт-мидл это **глубже, чем нужно**. Знать, что в Node не всё совпадает с браузером — достаточно.

## Что важно вынести

- **JS однопоточный.** Стек один. Между макротасками — очередь, не параллельность.
- **Web API не часть JS.** Таймеры, fetch, события — окружение. JS-движок не «ждёт».
- **Две очереди:** микротаски (`Promise.then`, `queueMicrotask`, `MutationObserver`) и макротаски (`setTimeout`, события, I/O).
- **Микротаски всегда раньше.** Между двумя макротасками очередь микротасок опустошается **полностью**.
- **`await` = `.then`.** Код после `await` — микротаска.
- **`Promise.then` вызывается раньше `setTimeout(0)`** — это типовой собесовский тест.
- **Microtask starvation:** бесконечная цепочка микротасок может задержать макротаски и рендер. Реальный риск в плохо написанных промисных цепях.
- **`requestAnimationFrame`** — для анимаций, синхронизирован с рендером браузера.
