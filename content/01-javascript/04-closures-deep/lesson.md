# Замыкания на уровне мидл+

Замыкания — самая частая тема на собесах по JS. И самая часто пересказанная неправильно: «функция помнит переменные», «доступ к внешней области» — это верно, но не объясняет **почему** и **как**. Цель урока — чтобы ты могла объяснить замыкания через **механику движка**, а не через метафоры.

> Ключевая мысль: замыкание — это не «фича», которую разработчики JS специально добавили. Это **естественное следствие** того, как устроен лексический scope в языке с функциями первого класса.

## Сначала — что такое scope

**Scope** (область видимости) — правила, по которым движок ищет переменные.

```js
function outer() {
  const x = 1;
  function inner() {
    console.log(x); // где движок ищет x?
  }
  inner();
}
```

JS использует **лексический scope** (его ещё называют статическим): где функция **написана** в коде — там она и ищет переменные.

Альтернатива — динамический scope: «ищу там, откуда меня вызвали». JS так не делает. Bash и старый Perl делали — и это создавало кошмар при отладке.

Лексический scope означает: глядя на код, ты можешь сказать, какие переменные доступны функции, **не зная, кто и откуда её вызовет**. Это критично для понимания замыканий.

## Что такое замыкание (определение через scope)

**Замыкание = функция + её лексический scope, сохранённый после того как внешний контекст завершился.**

```js
function makeCounter() {
  let count = 0;
  return function () {
    count++;
    return count;
  };
}

const counter = makeCounter();
counter(); // 1
counter(); // 2
```

Что произошло:
1. `makeCounter()` запустилась, создала локальную переменную `count`.
2. Вернула внутреннюю функцию.
3. `makeCounter()` завершилась — её **stack frame** ушёл.
4. Но `count` **не удалился**.

**Почему `count` не удалился:** внутренняя функция, ссылаясь на `count`, держит ссылку на **environment** (область переменных) внешней функции. Пока есть ссылка — garbage collector не освобождает память.

То есть замыкание — это **связка двух вещей**:
- Функция (её код).
- Environment, в котором функция была создана (переменные, к которым она имела доступ).

После `return` функция «уносит с собой» свой родительский environment. Этот механизм встроен в язык. Ты не пишешь его — он работает сам.

## Как движок это реализует: Environment Record

Каждый раз при вызове функции движок создаёт **Lexical Environment** — структуру с двумя частями:
- **Environment Record** — словарь имя → значение (локальные переменные функции).
- **Outer reference** — ссылка на родительский environment.

Когда внутри функции встречается имя `count`:
1. Сначала смотрим в свой Environment Record.
2. Не нашли — идём по `outer` ссылке к родительскому.
3. Так до глобального scope.

Это и есть **scope chain**. Когда внутренняя функция возвращается из внешней, она держит ссылку на родительский environment **через `outer`**. GC не может его удалить, пока есть эта ссылка.

> В терминах спецификации это `[[Environment]]` — internal slot функции. Можно не запоминать терминологию, но знать что замыкание реализовано через **ссылочные** связи, а не через копирование, — важно для понимания утечек памяти.

## Классическая ловушка: var в цикле

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Что выведет?
```

Ответ: `3 3 3`. Не `0 1 2`.

Объяснение:
- `var` имеет **функциональный scope** (или глобальный, если не в функции). Одна переменная `i` на весь цикл.
- К моменту выполнения колбэков цикл уже закончился, `i === 3`.
- Все три замыкания держат ссылку на **одну и ту же** `i`.

Фикс через `let`:
```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 0 1 2
```

`let` имеет **блочный scope** — каждая итерация создаёт **новую** переменную `i`. Каждое замыкание захватывает свою.

Это типовой собесовский вопрос. Если спросят «почему `var` так себя ведёт?» — отвечай через scope: одна переменная на весь функциональный scope vs новая на итерацию.

### Фикс через IIFE (исторически)

До `let` (ES6 — 2015 год) ту же задачу решали через IIFE — Immediately Invoked Function Expression:

```js
for (var i = 0; i < 3; i++) {
  (function (j) {
    setTimeout(() => console.log(j), 0);
  })(i);
}
```

Каждый вызов IIFE создаёт **новый** функциональный scope с **новой** переменной `j` (которая получает копию текущего `i`). Замыкание захватывает `j` своей итерации.

> На собесе про IIFE спрашивают как историческую данность. Сегодня используется `let`/`const`. Но знание IIFE — маркер «понимаю замыкания, а не просто использую современный синтаксис».

## Замыкание как способ создать приватность

В JS нет (до недавнего времени) приватных полей класса. Замыкания дают способ обойти:

```js
function makeAccount(balance) {
  return {
    deposit(amount) { balance += amount; },
    withdraw(amount) { balance -= amount; },
    getBalance() { return balance; },
  };
}

const acc = makeAccount(100);
acc.balance; // undefined — извне доступа нет
acc.deposit(50);
acc.getBalance(); // 150
```

`balance` живёт в замыкании трёх методов. Извне нет способа его прочитать или изменить, кроме предоставленных методов. Это **модульный паттерн**, основа инкапсуляции в JS до классов.

> Сегодня для приватности есть `#privateField` в классах (с 2022). Но замыкания всё ещё используются — например, в React-хуках (`useState` под капотом держит замыкание над текущим значением).

## React-хуки — это замыкания

```js
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

Каждый рендер компонента — **новый вызов** функции `Counter`. Создаются новые `count`, новый `handleClick`. Замыкание `handleClick` захватывает `count` **этого** рендера.

Отсюда знаменитая ловушка **stale closure**:

```js
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count); // всегда 0!
    }, 1000);
    return () => clearInterval(id);
  }, []); // пустой массив — эффект запустился один раз

  // ...
}
```

`useEffect` запустил `setInterval` с замыканием над `count` **первого** рендера. Этот `count === 0` и останется навсегда — следующие рендеры не пересоздают `setInterval`.

Фикс: убрать пустой массив зависимостей (но тогда interval будет пересоздаваться каждый рендер) или использовать функциональный setState: `setCount(c => c + 1)`.

> Понимание stale closure — обязательное для мидла на React. Спрашивают часто.

## Утечки памяти через замыкания

Поскольку замыкание держит ссылку на **весь** родительский environment — оно может удерживать вещи, которые тебе не нужны:

```js
function setup() {
  const hugeArray = new Array(1_000_000).fill("data");
  const id = "user-123";

  document.getElementById("btn").addEventListener("click", () => {
    console.log(id); // используем только id
  });
  // но замыкание держит и hugeArray, потому что они в одном scope
}
```

Замыкание захватывает **environment**, а не отдельные переменные. Даже если функция использует только `id`, она держит ссылку на environment, в котором есть `hugeArray`. Пока обработчик клика жив — `hugeArray` не освободится.

**Фикс — изолировать:**
```js
function setup() {
  const hugeArray = new Array(1_000_000).fill("data");
  doSomethingWith(hugeArray);

  const id = "user-123";
  setupHandler(id); // вынесли в отдельную функцию — её замыкание не видит hugeArray
}

function setupHandler(id) {
  document.getElementById("btn").addEventListener("click", () => {
    console.log(id);
  });
}
```

> На собесе на тему «как искать утечки в React-приложении» — упоминание stale closures и не-снимаемых обработчиков с замыканиями — сильный ответ.

## Карри и частичное применение

Карри (currying) — техника превращения функции от нескольких аргументов в цепочку функций от одного, использует замыкания:

```js
function add(a) {
  return function (b) {
    return a + b;
  };
}

const add5 = add(5);
add5(3); // 8
add5(10); // 15
```

`add5` — это замыкание над `a = 5`. Можно создать сколько угодно «специализированных» функций из одной общей. Используется в FP-стиле (Ramda, lodash/fp).

Близкое понятие — **partial application** (частичное применение): фиксируем часть аргументов, остальные получаем позже.

## Что важно вынести

- **Замыкание = функция + её лексический environment**, сохранённый после завершения внешнего контекста. Это не магия, это следствие лексического scope.
- **Реализация через ссылки:** функция держит `[[Environment]]` → родительский Environment Record → дальше по цепи. GC не освобождает, пока есть ссылка.
- **`var` в цикле даёт одну переменную на весь функциональный scope** — все замыкания видят финальное значение. **`let` создаёт переменную на каждую итерацию** — каждое замыкание захватывает свою.
- **IIFE** — старый способ создать новый scope, до `let`. Знать как историю.
- **Stale closure в React** — замыкание над `state` старого рендера. Лечится свежими зависимостями или функциональным `setState`.
- **Утечки памяти** — замыкание держит **весь environment**, не только используемые переменные. Изолировать через выделение функций.
- **Карри / partial application** — практическое применение замыканий в FP.
