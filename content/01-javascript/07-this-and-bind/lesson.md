# this, call/apply/bind и стрелочные функции

`this` — самая частая ловушка JS. Её спрашивают на каждом мидл-собесе, обычно в формате «что выведет этот код». Тебе нужны не догадки, а **4 правила**, по которым движок определяет `this` в любом вызове. Этот урок — про них и про то, как `call`/`apply`/`bind` дают контроль, а стрелочные функции — отменяют игру.

> Главная мысль: `this` определяется **не местом, где функция объявлена**, а **способом, которым её вызвали**. Это противоположно тому, как работает лексический scope. Стрелочные функции возвращают `this` в лексические правила — поэтому в них «нет своего this».

## 4 правила определения this

Когда движок встречает вызов функции `fn(...)`, он смотрит на **форму вызова** и выбирает одно из четырёх правил. Идут по приоритету: новый > явное > неявное > дефолт.

### 1. New binding (через `new`)

```js
function User(name) {
  this.name = name;
}
const u = new User('Лиза');
// this внутри User — это новый объект {}
```

Когда функцию вызывают через `new`:
- Создаётся новый пустой объект.
- Этот объект становится `this` внутри функции.
- Функция выполняется, обычно записывает в `this` поля.
- Возвращается этот новый объект (если функция явно не вернула другой объект).

Это и есть «конструктор». ES6-классы — синтаксический сахар над этой механикой.

### 2. Explicit binding (`call`/`apply`/`bind`)

```js
function greet() {
  console.log(this.name);
}
const user = { name: 'Лиза' };

greet.call(user);        // 'Лиза'
greet.apply(user);       // 'Лиза'
const bound = greet.bind(user);
bound();                 // 'Лиза'
```

**`call(thisArg, ...args)`** — вызывает функцию **с указанным `this`** и аргументами через запятую.

**`apply(thisArg, args[])`** — то же самое, но аргументы массивом.

**`bind(thisArg)`** — **не вызывает**, а **возвращает новую функцию** с зафиксированным `this`. Удобно для колбэков.

Мнемоника: **C**all = **C**ommas, **A**pply = **A**rray, **B**ind = **B**ound function.

### 3. Implicit binding (через точку)

```js
const user = {
  name: 'Лиза',
  greet() {
    console.log(this.name);
  },
};
user.greet();   // 'Лиза'

const fn = user.greet;
fn();           // undefined (или throw в strict mode)
```

Когда функцию вызывают **через точку или скобки доступа**, `this` — это объект **слева от точки**.

`user.greet()` — `this = user`.
`fn()` — нет точки, неявное связывание не работает, идём к правилу 4.

Эта потеря `this` — **классическая ловушка**. Колбэки страдают:

```js
setTimeout(user.greet, 0);
// user.greet вырывается из контекста, this теряется
// внутри greet → this.name → undefined
```

Решение: `setTimeout(() => user.greet(), 0)` или `setTimeout(user.greet.bind(user), 0)`.

### 4. Default binding (дефолтный)

Если ни одно из выше не сработало:
- **В strict mode** — `this = undefined`.
- **Без strict mode** — `this = globalThis` (`window` в браузере, `global` в Node).

```js
function fn() {
  console.log(this);
}
fn();  // window/global (или undefined в strict)
```

Современный код в модулях ESM **всегда strict mode**, так что `this = undefined`. В классе — тоже строго.

## Стрелочные функции — особая порода

Стрелочные функции **не имеют собственного `this`**. Они берут `this` из **лексического окружения** — то есть из места, где **объявлены**.

```js
const user = {
  name: 'Лиза',
  greet: () => {
    console.log(this.name);
  },
};
user.greet();
// this = window/undefined, потому что стрелочная функция объявлена
// в глобальном scope (не внутри метода), и взяла его this
```

Сравни:

```js
const user = {
  name: 'Лиза',
  greet() {
    setTimeout(() => {
      console.log(this.name);   // 'Лиза' ✅
    }, 0);
  },
};
user.greet();
```

Тут `greet` — обычный метод. При `user.greet()` его `this = user` (правило 3). Внутри `setTimeout` — стрелочная функция, **наследует `this` из greet**. Поэтому `this.name === 'Лиза'`.

Это и есть «зачем стрелочные функции» — они избавляют от `bind` и `var self = this` в колбэках.

### Где НЕ использовать стрелочные

1. **Методы объекта**, если нужно `this = объект` (правило выше)
2. **Конструкторы**: `new (() => {})` → `TypeError`
3. **Обработчики событий в DOM** через `addEventListener`, если хочешь `this = element`:
   ```js
   button.addEventListener('click', function () {
     this // → button ✅
   });
   button.addEventListener('click', () => {
     this // → внешний контекст ❌
   });
   ```

## Контекст в React — практика

В классовых компонентах `this` критичен. С функциональными — почти не нужен (хуки используют замыкания, не `this`). Поэтому современный React этот болевой паттерн закрыл — но на собесе про классы всё ещё спрашивают.

```js
class Counter extends React.Component {
  state = { count: 0 };

  handleClick() {
    // this = undefined, потому что onClick вызывается без точки
    this.setState({ count: this.state.count + 1 });  // TypeError
  }

  render() {
    return <button onClick={this.handleClick}>+</button>;
  }
}
```

Три фикса:
1. `onClick={this.handleClick.bind(this)}` — `bind` в рендере. **Минус**: новый bound function на каждый рендер → нарушение мемоизации.
2. `handleClick = () => { ... }` — стрелочная функция как поле класса. **Плюс**: `this` наследуется из конструктора (т.е. экземпляра).
3. `bind` в конструкторе: `this.handleClick = this.handleClick.bind(this)`. Старый паттерн.

Современный код — функциональный + хуки. Все три фикса — история.

## `call.bind(call)` и подобные трюки

На собесах любят показать что-нибудь странное:

```js
const slice = Array.prototype.slice;
const arr = slice.call({ 0: 'a', 1: 'b', length: 2 });
// ['a', 'b']
```

Что произошло: `slice` — метод массива. Его `this` обычно — массив. Через `call` мы передали псевдомассив (объект с `length`), и `slice` отработал на нём как на массиве. Это паттерн «**заимствование метода**» (method borrowing).

Раньше так конвертировали `arguments` и NodeList в массив:
```js
Array.prototype.slice.call(arguments);  // → [...args]
// Современный аналог:
[...arguments]
```

Знать **не для использования**, а для понимания чужого кода и собесового вопроса «как раньше делали без spread».

## Стрелочные функции и `arguments`

Ещё одна особенность: у стрелочных нет своего `arguments`.

```js
function fn1() {
  console.log(arguments);  // [Arguments] { 0: 'a' }
}
fn1('a');

const fn2 = () => {
  console.log(arguments);  // ReferenceError или unexpected
};
fn2('a');
```

В стрелочных используют rest: `(...args) => console.log(args)`. На практике — без проблем, но на собесе могут спросить.

## Сводная таблица: что у стрелочной нет

| Что | Обычная функция | Стрелочная |
|---|---|---|
| Свой `this` | Да (определяется вызовом) | **Нет** (берёт из лексического scope) |
| Свой `arguments` | Да | **Нет** |
| Можно вызвать через `new` | Да | **Нет** |
| Свой `super` | Да (в методе класса) | **Нет** (берёт из родителя) |
| `prototype` | Да | **Нет** |

«Нет» в стрелочной всегда означает «берёт из окружения». Эта симметрия запоминается легко.

## Что важно вынести

- **4 правила `this`** по приоритету: new > call/apply/bind > точка/скобка > дефолт. На собесе называй их.
- **Потеря контекста**: метод, вырванный из объекта (`const fn = obj.method; fn()`), теряет `this`. Лечится `bind`, стрелочной или `() => obj.method()`.
- **Стрелочные функции** не имеют своего `this/arguments/super/prototype`. Это и плюс (нет ловушек), и минус (нельзя как метод, нельзя как конструктор).
- **`call(thisArg, ...args)`** — вызвать сразу. **`apply(thisArg, [args])`** — то же, но массивом. **`bind(thisArg)`** — отложенный вызов, возвращает новую функцию.
- **Method borrowing** через `call` — исторический паттерн для работы с псевдомассивами. Сегодня заменён spread/rest.
- **Современный React (хуки)** обходит проблему `this` через замыкания — поэтому в новом коде она реже встречается, но классовые компоненты ещё в legacy.
