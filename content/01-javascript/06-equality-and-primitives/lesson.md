# Equality и примитивы: == vs === без воды

«Используй `===` всегда» — самый частый совет про сравнения в JS. Он работает, но не объясняет почему. На собесе спрашивают не «что использовать», а «**что под капотом** делает `==` и где оно ловит». Этот урок — про механику.

> Ключевая мысль: в JS существует **четыре** способа сравнения, и каждый предназначен для своего случая. Их разница не в производительности, а в **правилах приведения типов и обработки крайних случаев** (NaN, ±0, объекты).

## Сначала — какие сравнения есть

| Оператор | Имя в спецификации | Что делает |
|---|---|---|
| `==` | Loose equality | Сравнивает с **приведением типов** |
| `===` | Strict equality | Сравнивает **без приведения**, но с двумя странностями |
| `Object.is(a, b)` | SameValue | Как `===`, но без странностей |
| `[a].includes(b)` / `Array.prototype.indexOf` | SameValueZero | Как `Object.is`, но `NaN === NaN` |

На собесе достаточно знать первые три. Четвёртое — для эрудиции.

## `===` (strict equality)

Правила в одном абзаце:
1. Если типы разные — `false`.
2. Если оба `null` или оба `undefined` — `true`.
3. Если оба числа — обычное сравнение, **но** `NaN !== NaN` и `+0 === -0`.
4. Если оба строки — посимвольно одинаковые → `true`.
5. Если оба объекта — **только** если это **одна и та же ссылка**.

```js
1 === 1                  // true
'a' === 'a'              // true
null === null            // true
undefined === undefined  // true
1 === '1'                // false (разные типы)

NaN === NaN              // false  ← странность №1
+0 === -0                // true   ← странность №2

{a:1} === {a:1}          // false (разные объекты)
const o = {a:1}; o === o // true (одна ссылка)
```

**Странность №1: `NaN !== NaN`.** Почему так: `NaN` означает «результат, который нельзя представить числом». Два разных «нельзя» — не обязательно одно и то же. Это историческая норма IEEE 754 (стандарт чисел с плавающей точкой), и JS его соблюдает.

**Странность №2: `+0 === -0`.** Тоже из IEEE 754: знак у нуля есть, но математически они равны.

Эти две странности существуют в `===`, потому что он реализует **строгое равенство в смысле сравнения значений**, а не «алгоритмическое тождество». Когда тождество нужно — есть `Object.is`.

## `Object.is(a, b)` — без странностей

Появился в ES2015. Работает как `===`, но **починен**:
- `Object.is(NaN, NaN)` → `true`
- `Object.is(+0, -0)` → `false`

Остальное — как у `===`. Поэтому когда тебе нужно «**ровно то же самое значение**» — `Object.is`. На практике используют для:
- Сравнения, которое должно вернуть `true` для `NaN === NaN` (фильтрация массива с NaN)
- React: `Object.is` под капотом в `Object.is(prev, next)` для сравнения state в `useState`

**Запоминалка:** `===` — IEEE-friendly. `Object.is` — JS-friendly.

## `==` (loose equality)

«Не используй» — банально и верно. Но на собесе спросят: **почему**.

Алгоритм `==` приводит типы по сложным правилам:
1. Если типы совпадают — как `===`.
2. `null == undefined` → `true`. Между собой только. С остальным — `false`.
3. Число и строка → строка приводится к числу.
4. Boolean → приводится к числу (`true → 1`, `false → 0`), потом сравниваем.
5. Объект и примитив → объект через `valueOf()` или `toString()` к примитиву.

Самые ядовитые случаи:

```js
0 == ''           // true  ('' → 0, потом 0 == 0)
0 == false        // true  (false → 0)
'' == false       // true  (false → 0, '' → 0)
[] == false       // true  ([] → '' → 0)
[] == ![]         // true  (!![] → false → 0, [] → 0)
'0' == false      // true  ('0' → 0, false → 0)

null == undefined // true  (специальное правило)
null == 0         // false (специальное: null НЕ приводится к числу в ==)
null >= 0         // true  (НО >= приводит null к 0 — другой оператор, другие правила!)
```

Видишь, насколько хрупко. Логику `==` нужно знать **только** для того, чтобы понимать чужой код, который её использует. Свой код пишем на `===`.

### Исключение: `x == null`

Единственный паттерн, где `==` используют сознательно: проверка «null или undefined одновременно».

```js
if (value == null) { ... }
// эквивалент:
if (value === null || value === undefined) { ... }
```

Короче. В React-сообществе допускается, в TS — линтер обычно ругается, но фикс есть: `eslint-disable-next-line` рядом с комментарием «why».

## Примитивы vs объекты — глубже

В JS **7 примитивов**:
- `string`, `number`, `boolean`, `null`, `undefined`, `symbol`, `bigint`

Всё остальное — **объекты**: `{}`, `[]`, `function`, `Date`, `Map`, `Set`, и т.д.

Главное различие в сравнении:
- **Примитивы** сравниваются **по значению**: `5 === 5` это true, потому что оба «5».
- **Объекты** сравниваются **по ссылке**: два разных объекта с одинаковыми полями — это разные объекты.

```js
const a = 'hello';
const b = 'hello';
a === b              // true: примитивы, значения совпадают

const x = { id: 1 };
const y = { id: 1 };
x === y              // false: разные объекты
const z = x;
x === z              // true: одна ссылка
```

Это и есть причина, почему React требует **новый** объект для триггера ре-рендера в `useState`:
```js
const [user, setUser] = useState({ name: 'Лиза' });

// Так НЕ сработает:
user.name = 'Лиза';
setUser(user);  // ссылка та же → React: «ничего не изменилось»

// Так сработает:
setUser({ ...user, name: 'Лиза' });  // новый объект → новая ссылка
```

## Глубокое сравнение объектов — отдельная история

`===` для объектов — это про ссылки. Чтобы сравнить **содержимое**, нужно либо:
1. Вручную: `a.id === b.id && a.name === b.name`
2. Сериализовать: `JSON.stringify(a) === JSON.stringify(b)` (хрупко: разный порядок ключей, undefined, Date, циклы)
3. Использовать утилиту: `lodash.isEqual`, `dequal`

В нативном JS глубокого equals **нет**. Если бы он был — тебе бы пришлось спорить, как сравнивать функции и Date'ы. Поэтому язык не решил за тебя.

## Сводная таблица

| Случай | `==` | `===` | `Object.is` |
|---|---|---|---|
| `1 == 1` | true | true | true |
| `1 == '1'` | **true** | false | false |
| `null == undefined` | **true** | false | false |
| `NaN == NaN` | false | false | **true** |
| `+0 == -0` | true | true | **false** |
| `{a:1} == {a:1}` | false | false | false |
| `[] == false` | **true** | false | false |

Чёрным жирным — «странные» места. Зная их — на собесе можно отвечать без шпаргалки.

## NaN: как проверить «это NaN»

`NaN !== NaN` (странность `===`). Поэтому:

```js
const x = NaN;
x === NaN              // false ❌
x !== x                // true  ✅ (классический трюк)
Number.isNaN(x)        // true  ✅ (правильный способ)
isNaN(x)               // true  ⚠️ (глобальный isNaN тоже приводит типы — небезопасен)
```

**`Number.isNaN`** vs **`isNaN`** — частый собесовский вопрос:
```js
isNaN('foo')          // true: 'foo' → NaN → isNaN → true (двойная конверсия!)
Number.isNaN('foo')   // false: 'foo' это не NaN, а строка
```

`Number.isNaN` — строгая проверка: «является ли это **значением** NaN». `isNaN` — мягкая: «**могло бы** это стать NaN при приведении к числу». Используй `Number.isNaN`.

## Что важно вынести

- **`===`** — обычное сравнение, но `NaN !== NaN` и `+0 === -0` (IEEE 754).
- **`Object.is`** — как `===`, но **`NaN === NaN`** и **`+0 !== -0`**.
- **`==`** приводит типы — ядовито в крайних случаях (`'' == 0`, `[] == false`). Не используй, кроме паттерна `x == null`.
- **Примитивы** сравниваются по значению, **объекты** — по ссылке. Это и причина паттерна «новый объект для setState».
- **Глубокого `===` в JS нет** — нужен `JSON.stringify` (хрупко) или утилита (`dequal`/`lodash.isEqual`).
- **`Number.isNaN`** — строгая проверка, **`isNaN`** — мягкая с приведением. На собесе называй разницу.
