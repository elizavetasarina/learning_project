# Архитектура fullstack-приложения

Ты написал фронтенд, подключил базу данных, настроил аутентификацию. Но можешь ли ты объяснить, как все эти куски связаны? На собесе мидл+ это один из самых частых вопросов: «расскажите архитектуру вашего проекта». Этот урок — подготовка к такому ответу.

> Ключевая идея: архитектура — это не «какие файлы где лежат», а **как данные текут через систему**: от клика пользователя до записи в базе и обратно.

## Два мира: сервер и клиент

В Next.js App Router каждый компонент по умолчанию — **серверный**. Он выполняется на сервере (или при билде), имеет доступ к файловой системе и переменным окружения, но не может использовать `useState`, `useEffect` или обращаться к `window`.

Когда нужна интерактивность (клики, формы, анимации, API браузера) — добавляем `"use client"` в начало файла. Этот компонент отправляется в браузер и выполняется там.

> Правило: серверный компонент может импортировать клиентский, но не наоборот. Данные текут сверху вниз — серверный готовит, клиентский отображает.

Пример из нашего проекта:

```
page.tsx (серверный)        → читает уроки из файлов
  └── LessonsList (клиентский) → подтягивает прогресс из API
        └── useProgress()      → хук с useState + useEffect
```

Зачем разделять? Серверный компонент читает файлы **быстро** (без сетевого запроса). Прогресс нужен из API (нужен `window.Telegram` для аутентификации) — это клиентская задача. Каждый делает то, что умеет лучше.

## Путь запроса: от клика до базы

Когда пользователь завершает тест, происходит цепочка:

```
1. Браузер (quiz-view.tsx)
   → saveProgress("01-types", 80)
   → fetch POST /api/progress
     Headers: X-Init-Data: "user=...&hash=abc..."
     Body: { lessonSlug: "01-types", score: 80 }

2. Сервер (route.ts)
   → authenticateRequest(request)
     → validateInitData(initData)  — HMAC проверка
     → prisma.user.upsert()        — создать/найти юзера
   → INSERT ON CONFLICT (SQL)      — сохранить прогресс
   → return { progress: { bestScore: 80 } }

3. Браузер
   → получает ответ, показывает итоговый экран
```

Каждый шаг имеет свою зону ответственности. Клиент не знает про SQL. Сервер не знает про UI. Между ними — HTTP-запрос с JSON.

## Аутентификация: как сервер знает, кто ты

Telegram Mini App передаёт данные пользователя в `window.Telegram.WebApp.initData`. Но этим данным нельзя доверять напрямую — их может подделать любой через DevTools.

Telegram **подписывает** данные секретным ключом (производным от токена бота). Вместе с данными приходит `hash` — подпись. Сервер повторяет вычисление и сравнивает:

```
1. Берём все параметры кроме hash
2. Сортируем по алфавиту, склеиваем через \n
3. secret = HMAC-SHA256("WebAppData", BOT_TOKEN)
4. computed_hash = HMAC-SHA256(secret, data_check_string)
5. computed_hash === hash? → данные подлинные
```

Это **stateless-подход**: при каждом API-запросе клиент передаёт initData в заголовке `X-Init-Data`, сервер проверяет подпись. Никаких сессий, cookie, таблиц сессий в БД. Каждый запрос самодостаточен.

> Альтернатива — сессионный подход: проверяем initData один раз, создаём JWT или сессию, дальше работаем по ней. Это сложнее (нужен механизм инвалидации), но экономит вычисления при частых запросах. Для Mini App stateless — лучший выбор.

## HMAC: подпись, а не шифрование

HMAC (Hash-based Message Authentication Code) — это **не шифрование**. Данные передаются открытым текстом. HMAC гарантирует только одно: данные не были изменены после подписания.

Обычный хеш (SHA256) может вычислить любой — он не требует секрета. HMAC добавляет секретный ключ: без знания ключа (токена бота) невозможно создать корректный hash.

Важная деталь: при сравнении подписей используется `timingSafeEqual` вместо `===`. Обычное сравнение строк прекращается при первом несовпадении — атакующий может измерять время ответа и подбирать hash побайтово. `timingSafeEqual` тратит одинаковое время независимо от совпадения — это защита от **timing attack**.

## ORM и база данных

**ORM** (Object-Relational Mapping) — прослойка между кодом и SQL. Вместо написания SQL-запросов вручную ты пишешь:

```ts
prisma.user.findUnique({ where: { telegramId: 123n } })
// → SELECT * FROM users WHERE telegram_id = 123
```

Prisma генерирует TypeScript-типы из схемы БД — если колонка называется `firstName`, TypeScript не даст написать `user.firstname` (с маленькой буквы). Это ловит ошибки на этапе компиляции, а не в runtime.

> Когда ORM недостаточно: если нужна логика, которую ORM не умеет выразить (например, `GREATEST()` для выбора максимума), спускаемся на raw SQL. Prisma поддерживает это через `$queryRawUnsafe`. Это нормальная практика — ORM для 90% запросов, raw SQL для остальных.

### Upsert — вставка или обновление

Частая задача: «создай запись если не существует, иначе обнови». Два подхода:

**Prisma ORM:**
```ts
prisma.user.upsert({
  where: { telegramId: 123n },
  create: { telegramId: 123n, firstName: "Лиза" },
  update: { firstName: "Лиза" },
})
```

**PostgreSQL SQL:**
```sql
INSERT INTO users (telegram_id, first_name)
VALUES (123, 'Лиза')
ON CONFLICT (telegram_id)
DO UPDATE SET first_name = EXCLUDED.first_name
```

`EXCLUDED` — специальная ссылка на данные, которые «пытались вставить». Если конфликт — они доступны для обновления.

## Синглтон и connection pooling

**Проблема:** в dev-режиме Next.js при hot reload пересоздаёт модули. Каждый `new PrismaClient()` — новое подключение к БД. За час: 50+ подключений → база отказывает.

**Решение — синглтон:** храним один экземпляр в `globalThis` (объект, который переживает hot reload). При перезагрузке модуля код видит существующий экземпляр и не создаёт новый.

**Connection pooling** — похожая идея на уровне БД. Supabase использует pgBouncer: вместо того чтобы каждый запрос создавал новое TCP-подключение (дорого: DNS → TCP handshake → TLS → auth), pgBouncer держит пул готовых подключений и переиспользует их.

## Структура проекта: что где лежит

```
lib/           — серверная логика (не React)
  prisma.ts       синглтон БД
  auth.ts         HMAC + upsert юзера
  telegram-auth.ts  криптография валидации
  lessons.ts      чтение markdown-файлов
  api-client.ts   клиентские fetch-обёртки
  pluralize.ts    утилита склонений

hooks/         — React-хуки (клиентские, с useState/useEffect)
  use-telegram.ts   данные из Telegram SDK
  use-progress.ts   загрузка прогресса из API

app/api/       — API-эндпоинты (серверные)
  progress/route.ts  GET + POST прогресса

content/       — уроки и тесты (markdown + JSON)
```

Правило: `hooks/` — только функции с `use*`, которые используют React state. `lib/` — всё остальное: утилиты, серверная логика, API-клиент.

## Что важно вынести

На собесе по этой теме могут спросить:

- **«Как устроена аутентификация?»** — Telegram подписывает данные HMAC-SHA256, сервер проверяет подпись при каждом запросе (stateless). timingSafeEqual для защиты от timing attack.
- **«Почему stateless, а не сессии?»** — В Mini App Telegram всегда передаёт свежий initData. Сессии — лишняя сложность без выигрыша. Tradeoff: чуть больше вычислений (HMAC на каждый запрос), зато нет state на сервере и горизонтальное масштабирование из коробки.
- **«Зачем ORM если можно писать SQL?»** — Типобезопасность, автодополнение, миграции, защита от SQL-инъекций. Но когда ORM не хватает — спускаемся на raw SQL (ON CONFLICT, GREATEST).
- **«Что такое connection pooling?»** — Переиспользование TCP-подключений к БД через пул (pgBouncer). Экономит ресурсы, ускоряет запросы.
- **«Серверные vs клиентские компоненты?»** — Серверные: доступ к файлам, env, БД. Клиентские: интерактивность, браузерные API. Данные текут от сервера к клиенту через пропсы.
