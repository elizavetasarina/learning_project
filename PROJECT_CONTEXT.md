## 2. Что это за приложение

**Формат:** Telegram Mini App

**Кратко:** платформа для самостоятельной подготовки к собесам фронтенд-разработчика. Теория в виде интерактивных уроков + тесты разных форматов + система прогресса + spaced repetition.

## 3. Стек и архитектурные решения

### Стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Frontend | Next.js 14+ (App Router) | Знаю, серверные компоненты для контента |
| Язык | TypeScript (strict mode) | Стандарт мидл+ уровня |
| Стили | Tailwind CSS | Знаю, быстрая разработка |
| Markdown | react-markdown + remark-gfm + rehype-highlight | Контент в .md, подсветка кода |
| State | React Context на старте → Zustand на 3 неделе | От простого к сложному осознанно |
| Backend | Next.js API Routes | Не плодим сервисы без нужды |
| БД | Postgres через Supabase (free tier) | Бесплатно, проще Railway |
| ORM | Prisma | Учит хорошим паттернам, спрашивают на собесах |
| Деплой | Vercel | Бесплатно, нативная интеграция с Next.js |
| Аутентификация | Telegram initData + HMAC валидация на бэке | Учу безопасность на практике |

### Структура контента

**Контент хранится как файлы в репозитории, БД хранит только прогресс пользователя.**

Почему такой выбор:
- Markdown писать быстрее, чем заполнять формы или TS-объекты
- Git-версионирование контента
- Можно попросить AI сгенерировать готовый .md и просто положить в проект
- Профессиональный паттерн (так делают многие реальные обучающие платформы)
- На собесе можно рассказать про content-as-code

```
content/
  01-what-is-mini-app/
    lesson.md             # теория + frontmatter (title, description, order)
    quiz.json             # тестовые вопросы с пояснениями
  02-project-setup/
    lesson.md
    quiz.json
```

Решение (принято на неделе 1):
- Плоская структура, папка = урок (без промежуточного уровня blocks/)
- Frontmatter внутри lesson.md вместо отдельного meta.json
- quiz.json вместо questions.json (короче, понятнее)
- Без index.json — список уроков строится динамически из файловой системы
- Группировка по модулям (blocks/) добавляется позже, если уроков станет 15+

### Визуальная архитектура (принято на неделе 1)

**Дизайн-система:** описана в Design_System.md (вайб, цвета, типографика, компоненты).

**Цветовая стратегия:**
- Фон, текст, подсказки, бордеры — из CSS-переменных Telegram (подстраиваются под тему пользователя)
- Акцент (primary): #6366F1 (индиго) — свой цвет для кнопок, прогресс-баров, выделений
- Success/Error/Warning — свои (#10B981 / #EF4444 / #F59E0B)
- Итого: приложение вписывается в тему Telegram, но имеет свой характер через акцент

**Экраны и их состав:**

Главная (`/`):
- Приветствие + имя из Telegram
- WeekStreak — 7 кружков (Пн–Вс), заполненные = дни с активностью, текущий выделен. Рядом — счётчик "N дней подряд"
- Карточка "Продолжить" — последний незавершённый урок
- Мини-статистика: пройдено уроков / всего, лучший стрик
- Кнопка "Все уроки" → /lessons
- (Данные стрика/прогресса — заглушка до недели 2, когда появится БД)

Список уроков (`/lessons`):
- Карточки уроков (secondary-bg-color, padding 20px, radius 12px)
- В карточке: название, описание, кол-во вопросов, прогресс-бар
- Сортировка по order из frontmatter

Страница урока (`/lessons/[slug]`):
- Ссылка "← К урокам"
- Markdown-контент со стилизованными заголовками, списками, кодом
- Разделитель
- Тест (Quiz) — визуально отделён от теории

Тест (Quiz):
- Прогресс "Вопрос N из M"
- Choice-вопросы: карточки-варианты с бордером, при выборе — индиго, при ответе — зелёный/красный
- Input-вопросы: поле ввода, при ошибке — показ правильного ответа
- Пояснение после каждого ответа (всегда, правильный или нет)
- Кнопки 48px высотой, font 600

**Библиотека иконок:** Lucide React (единственная, 20px inline / 24px кнопки).

### Чего НЕ делаем (явный список)

Это важно — без этого списка проект будет разрастаться бесконечно:

- ❌ Админка для контента (контент = файлы в репозитории)
- ❌ Регистрация через email (только Telegram)
- ❌ Сложная ролевая модель (только обычные юзеры)
- ❌ Мультиязычность (только русский на старте)
- ❌ Тесты (unit/e2e) на ранних этапах — только когда логика стабилизируется
- ❌ Микрофронтенды, монорепо, сложные абстракции
- ❌ Custom UI library — используем Tailwind utility classes
- ❌ Серверный кэш Redis и подобное — Postgres достаточно

## 6. Правила работы с AI-ассистентами

### Общий принцип

**Я хочу понимать каждую строку кода, которая попадает в проект.** AI-ассистент — это инструмент обучения и ускорения, а не способ получить готовый код, который я не понимаю. Если на собесе меня попросят объяснить кусок этого проекта, я должна суметь — иначе проект бесполезен.

### Для Claude Code (при работе в проекте)

**Перед написанием кода:**
- Сначала объясни концепцию / архитектурное решение
- Покажи альтернативы и почему выбираешь именно этот вариант
- Только потом пиши код

**В процессе:**
- Перед каждым новым файлом — короткое объяснение его роли
- Комментарии в коде на русском, к нетривиальным местам
- Незнакомые мне паттерны или API объяснять перед применением
- НЕ добавлять зависимости без объяснения, зачем они
- НЕ создавать "на всякий случай" утилиты, тесты, абстракции
- НЕ молчаливо менять архитектуру — сначала обсудить

**После:**
- Краткий summary что сделано
- Что я могу проверить, что работает
- Какие 3-5 контрольных вопросов я могла бы задать себе для закрепления

**Стиль кода:**
- Функциональные компоненты, hooks
- Именованные экспорты (кроме страниц Next.js)
- Типы — отдельно, не inline в больших компонентах
- TypeScript strict, без `any`

### Для Claude-в-чате (роль преподавателя/ментора)

Контекст: я общаюсь с Claude в обычном чате как с ментором. Этот Claude помогает мне с теорией, разбирает мои ответы, задаёт вопросы, держит роадмап, поддерживает в моменты саботажа.

**Что важно от этого Claude:**
- Прямая обратная связь, без сахара — но с уважением
- Помнит про мой паттерн саботажа и не даёт уйти в избегание
- Объясняет теорию глубоко, на уровне мидл+
- Регулярно проверяет понимание вопросами
- Помогает оформлять выученное в .md-файлы для приложения

## 7. Антипаттерны (чего избегаем)

**В разработке:**
- "Сделай мне всё сразу" — нет, по шагам с объяснениями
- "Скопировал из StackOverflow, работает" — нет, понимаю или не использую
- "Добавим эту библиотеку на всякий случай" — нет, только под конкретную задачу
- Преждевременная оптимизация — сначала работает, потом красиво

**В обучении:**
- 5-часовые марафоны и потом 3 дня ничего — нет, регулярные 1-1.5 часа
- "Сначала наполню весь контент, потом буду учиться" — нет, контент рождается ИЗ учёбы
- Откликаться на 50 вакансий и игнорить ответы — нет, 5-10 в неделю с обязательным фоллоу-апом
- Подавать на западные позиции с минимальным английским — нет, сначала через СНГ-периметр

**В поиске работы (когда придёт время):**
- НЕ увольняться "чтобы сосредоточиться на поиске"
- НЕ соглашаться на первый оффер из-за тревоги
- НЕ откликаться на $3000+ позиции пока английский слабый — получишь отказы и подкрепишь "я ничего не могу"
